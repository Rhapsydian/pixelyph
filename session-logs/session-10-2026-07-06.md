## Phase 9: Palette/Layers/Style Overhaul

New `Palette.js` model (`colors`/`fills`/`styles` groups, replacing the old bare color array, with a `normalizePalette` migration shim for old saves) and matching store actions. `ColorAlphaInput.jsx` went through several iterations before landing on its final shape: a single checkerboard-backed swatch opening a `position: fixed` popover (hex text + R/G/B/A sliders+number boxes, escaping any scrollable ancestor like the gradient modal's own panel) with a "Color Picker" button that opens a fully custom, app-styled saturation/value + hue picker — not the native, unstyleable `<input type="color">` popup — including a real screen-sampling eyedropper via the `EyeDropper` API, which hides the picker modal during sampling so its own backdrop doesn't block the pick. New `GradientEditorModal.jsx` (shared between live layer-fill editing and palette-gradient authoring) and `FillSwatch.jsx` (one small SVG-based fill preview reused across the fill editor, the gradient modal, and both palette panels). `PalettePanel.jsx` reworked around the three groups, plus a new `ManageSwatchesModal.jsx` for reorder/rename, reachable from a Palette menu or the panel's own "Manage" button. Layer visibility moved from per-layer to per-frame (`Layer.js`, `Canvas.js`, `composeLayersSvg.js`, `projectFile.js` with a legacy-save migration), so different frames of an animation can each hide different layers, and a layer hidden in the active frame is now also effectively locked there. The default palette became PICO-8's 16 colors plus a true `#FFFFFF` white (PICO-8's own lightest tone is an off-white cream).

## Follow-on: Iterative Feedback on the Palette, Color Picker, and Gradient UI

The largest stretch of the session was rounds of direct feedback from using the result: capitalizing "Patterns" (before patterns were removed entirely — see below), moving swatch reordering/clearing out of the main panel into the new Manage Swatches modal, dropping a redundant quick-pick color row, centering and re-chroming the color-picker modal with a real "Select" button, hiding that modal during eyedropper use, swapping the fill swatch and save-to-palette button order, collapsing linear/radial gradient into one "Gradient" option in the fill-kind selector (the modal itself is where you actually pick between them), enlarging the gradient modal with a wide horizontal bar of draggable triangle "tabs" for each stop (capped-height, independently scrollable stop list so the modal itself never grows or scrolls), right-click-to-delete with a confirmation dialog and descriptive palette-panel copy, and replacing plain "Save to palette"/"Save style" text buttons with a hand-drawn arrow-into-palette icon. The SVG pattern fill kind — added earlier in this same phase — was tried, then removed outright (not just hidden) once it became clear that pasting raw SVG markup with no authoring or validation was a poor fit; logged to `BACKLOG.md` for a real tile-drawing UI later instead.

Two invalid-CSS bugs surfaced from this work and shared one root cause: `repeating-conic-gradient`/gradient-stop syntax using unitless angle/percentage values is invalid CSS, and when it's one layer of a multi-value `background-image`, the browser silently drops the *entire* declaration — not just that layer. This broke both the gradient modal's stop bar (meant to preview the live gradient as stops are dragged) and the color swatch's checkerboard backdrop; both were fixed by using valid syntax, and the stop bar now genuinely live-updates as a real `linear-gradient` regardless of which gradient type (linear/radial) is selected.

## Follow-on: Layout Fixes, Zoom Range, Fullscreen, Icon Redesign

The side panel's minimum width was raised (240→340px) after direct measurement showed the Layers row's opacity `%` box was wrapping to a second line below it; a missing `type="text"` on the layer-name input (the CSS attribute selector needs the literal attribute, not just the IDL default) was also fixed so it matches the opacity input's height. A gradient-swatch/"+"-button misalignment was root-caused to a global `button { height: var(--control-height) }` CSS default silently overriding component-level sizing on any swatch-wrapping button that didn't explicitly set its own `width`/`height` — fixed in three separate places (`PalettePanel.jsx`'s two swatch groups, `LayerStylePanel.jsx`'s gradient trigger, `ColorAlphaInput.jsx`'s default trigger) once the shared cause was recognized. Smaller asks: the zoom slider's minimum dropped from 4x to 1x; a fullscreen toggle button appeared in the header (new `useFullscreen.js` hook wrapping the standard Fullscreen API — one code path for both the web and Electron builds), mirrored in a new Window menu; and the hand-drawn "save to palette" icon was redrawn with real geometric separation between its arrow and palette blob (they previously overlapped by half a unit) and rendered larger.

## File Menu Restructuring: Export, Import, and a New Help Menu

The standalone "Export" top-level menu was removed in favor of a single "Export…" item under File, opening a new `ExportModal.jsx` — mode-aware: Draw mode gets a format selector (SVG/PNG/WebP/animated GIF), an active-frame-vs-whole-animation choice (shown only when it's actually meaningful — WebP has no animated path, GIF *is* the animated path), and a raster-scale selector; Glyph mode reuses `FontExportPanel.jsx` unmodified alongside a "export active glyph as SVG" action, replacing the old side-panel "Font" export tab entirely. The side panel's "Import" tab was removed; Import Image and Reference Image moved to the File menu, each opening its own settings modal (`ImportImageModal.jsx` / `ReferenceImageModal.jsx`), replacing the old always-visible `ImportImagePanel.jsx` (deleted outright — its two concerns were genuinely distinct: one paints imported pixels, the other sets a non-exported trace-over guide). Copy-as-SVG moved to the Edit menu and was relabeled "Copy SVG (for Illustrator, Inkscape, etc.)" since its old label didn't communicate what it was actually for. A new Help menu adds "About Pixelyph" (a modal reading the description/version straight out of `package.json`) and "Visit on GitHub," which needed a new Electron IPC handler (`shell.openExternal`) and a small `openExternalUrl` platform helper so the link opens the user's actual default browser — a new OS-level window from Electron, a new tab on the web — rather than a second in-app window loading the URL.

## Notes

- This session's actual code changes go well beyond the single `25aa3b0` commit — everything from the color-picker/gradient/palette follow-on rounds through the menu restructuring, backlog entries, and README update is still sitting uncommitted in the working tree, per this project's standing convention of only committing when explicitly asked.
- Worth remembering for next time: an invalid CSS gradient/multi-layer `background-image` value doesn't just fail to render *that* layer — the browser drops the whole declaration silently, with no console warning. Two separate features (a gradient preview bar, a checkerboard swatch backdrop) broke the same way for the same reason before the shared root cause was recognized.
- Also worth remembering: a global CSS default (`button { height }`, `input[type="text"] { height }`) silently wins over an unset component-level style rather than erroring — this bit three different swatch/trigger buttons independently before the pattern was recognized and checked for everywhere else it could apply.
