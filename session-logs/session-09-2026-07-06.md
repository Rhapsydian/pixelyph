## Plan-mode research and proposal

`WebSearch` was flaky this session (errored on most queries) and `WebFetch` can't render live single-page apps, so Piskel's and Aseprite's actual web editors were unreachable directly — but Aseprite's own docs (`workspace-layout`) explicitly name "Tool Bar" and "Context Bar" as separate, independently-movable panels, and state the tool bar specifically can't be moved to the top of the screen (vertical-only by design) — real confirmation, not just recalled knowledge, for the rail + context-bar split proposed below. Wireframe mockups were produced via the visualize tool to show the target layout before writing code.

The user iterated on the proposal directly rather than approving it outright: hand-rolled inline SVG icons instead of a new npm dependency (stock-style placeholders now, swappable for custom art later without touching call sites); a **vertical** left tool rail, not horizontal; a real **File/Edit/Export** menu bar instead of one bundled dropdown, with Edit surfacing Cut/Copy/Paste/Select All/Deselect as visible, clickable items (previously keyboard-shortcut-only with no visible control anywhere); and independent fill/stroke/effect alpha via 8-digit hex, which turned out to need zero model/export changes since SVG already accepts `#RRGGBBAA` directly.

## Follow-on: Centering, Dark Checkerboard, Resize Placement, Wheel Zoom

Four specific asks: canvas centered in its viewport, the transparency checkerboard matched to the wireframe's dark shades (was still light gray/white), the canvas/glyph resize control moved out of the header into the context bar, and scroll wheel wired to zoom. The wheel handler needed a native listener with `{ passive: false }` — React's synthetic `onWheel` is passive by default and silently ignores `preventDefault()`, which would otherwise scroll the page alongside zooming.

**Commit:** `521e557`

## Follow-on: Grid Visibility, Resizable Panels

The grid overlay was black at low opacity, effectively invisible against the new dark checkerboard — brightened to white at 40%. The frame strip and side panel needed to become resizable (frame strip: locked minimum height, adjustable via a drag handle; side panel: adjustable width) — built a small shared `useResizeDrag.js` hook for both. Making the frame strip a real bottom-docked panel surfaced a genuine layout bug: `.app-shell` only had `min-height: 100vh`, so once the frame strip claimed a fixed height, total content could exceed the viewport and the whole *page* scrolled instead of individual panels — fixed by giving `html`/`body`/`#root`/`.app-shell` a definite `height`, not just a minimum.

**Commit:** `62b69af`

## Follow-on: Control Consistency, Viewport Minimap, Grid Default

The largest single batch. Grid off by default. Buttons/text-inputs/selects unified to one `--control-height` token so mixed rows line up; a custom select chevron (matching the icon set) with left/right insets kept equal; number-input spinner arrows hidden. The frame strip needed real panel chrome (background/border) and a big enough minimum height to avoid triggering its own internal scrollbar — which surfaced a real bug: `flexWrap` on its controls row let content wrap onto multiple lines at narrow widths, inflating the height needed well past any fixed minimum; fixed with `nowrap` instead of continuing to guess at ever-larger minimums. Side panel's minimum width bumped up slightly. Biggest change: native canvas scrollbars removed entirely in favor of a self-managed pan — the SVG is absolutely positioned inside an `overflow: hidden` container, offset from a `pan` store field that (surprisingly) already existed in the model since early phases but had never actually been wired to anything, plus a `ResizeObserver`-measured `viewportSize`. New `ViewportPreview.jsx` renders a minimap above the side panel's tabs: a small full-canvas thumbnail, the relocated zoom slider, and — once zoom exceeds what's visible — a proportional, draggable rectangle for panning.

## Notes

- The preview tool has a real, reproducible quirk worth remembering next session: `preview_resize` with a custom width/height (as opposed to a named preset like `desktop`) sometimes renders screenshots with content squished into a corner while the actual DOM layout — confirmed via `getBoundingClientRect` — is correct at full size. Screenshots alone aren't reliable evidence at custom viewport sizes; cross-check with `preview_inspect`/`preview_eval` first.
- Two "obvious" interaction fixes this session initially looked broken during verification for the same underlying reason: reading DOM/store state in the same synchronous tick as a dispatched wheel or pointer event, before React had flushed its re-render. Not app bugs — timing issues in how the checks themselves were written. Re-querying after the synchronous script returned resolved both.
- The button-vs-label centering investigation is worth remembering as a pattern: rather than guessing at both possible culprits and patching both, verifying via direct measurement that plain `<button>` text was already correctly centered narrowed the real regression down to `<label>` elements specifically, before writing any fix.
