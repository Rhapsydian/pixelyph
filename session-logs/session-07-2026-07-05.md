## Phase 6: Electron Desktop Shell

**Scaffold:** `electron-vite` (not `vite-plugin-electron`, per the plan) with `electron/main/index.js` (BrowserWindow + IPC handlers) and `electron/preload/index.js` (contextBridge), configured via `electron.vite.config.mjs` to reuse the project's existing root `index.html`/`src/` layout rather than moving files under electron-vite's default `src/renderer/` convention. `platform.js` and `autosave.js` already had an `isElectron()` branch expecting a `window.pixelyph.{saveFile,openFile,writeAutosave,readAutosave,clearAutosave}` bridge (built ahead of time in an earlier phase) — Phase 6 was mostly filling in the main-process side of a contract that already existed on the renderer side.

**Deviations from the plan, all tooling/environment quirks rather than design gaps:**
- The renderer's `base` had to be `'./'` (relative) instead of the web build's GitHub Pages `/pixelyph/` prefix, since the packaged app loads `index.html` via `file://`, not a server root.
- electron-vite detected the project's `"type": "module"` and built main/preload as real ES modules, auto-shimming `__dirname` via `import.meta.dirname` — no CJS/ESM interop hack needed, unlike Phase 5's opentype.js surprise.
- In this environment, `npm install`'s postinstall didn't actually download the `electron` binary (`node_modules/electron/dist/` was empty) until `node node_modules/electron/install.js` was run manually — an environment quirk, not a code issue, but worth knowing if a future session sees Electron inexplicably fail to launch after a fresh install.

**Packaging:** `electron-builder` targeting Windows (NSIS installer), `npm run dist:win`.

**Testing:** per the plan's own stated policy, Phase 6 has no automated test suite — but "verified" here meant more than reading the code. A scripted Playwright `_electron` driver booted both `electron-vite dev` and the fully packaged/`asar`'d installer output, dispatched real synthetic pointer events to paint a pixel, and round-tripped Save → Open through the actual native-dialog IPC handlers and filesystem — the OS dialog itself was stubbed at the main-process level (no automation tool can click into a real Windows dialog), but everything on either side of it, including the fs write/read, ran for real. Autosave was confirmed writing to the app-data JSON file rather than IndexedDB. `node --test` stayed 197/197 throughout.

## Header Layout Bug: File Menu Overflowing the Window in Glyph Mode

User-reported: in Glyph mode, the File menu sat far off to the right and its dropdown overflowed past the window edge. Root cause: the header used `justify-content: space-between` across three children (branding, File menu, size control) — but `GlyphSizeControl` renders `null` until a glyph is selected, leaving only two children, which `space-between` then spread to the two extreme edges of the header. Fixed by dropping `space-between` for normal left-to-right flow, with the trailing size control pushed right via `margin-left: auto` on a wrapping `<span>` instead — verified in the preview for both Draw and Glyph (no glyph selected) layouts. Explicitly scoped as a quick positional fix only, per the user, with a fuller header/toolbar visual pass already on the books for Phase 8.

## Right-Click to Erase

Feature request: right-click should erase instead of paint for pencil, bucket fill, line, rectangle, and ellipse, without switching to the dedicated eraser tool.

- `SvgPixelEditor` captures which button started a gesture (`ctx.erasing = evt.button === 2`) once per pointer-down — `pointermove` always reports button `0` even while a button is held, so this can't be rechecked mid-drag.
- New `src/ui/draw/tools/toolColor.js` resolves that into the actual paint value (`null`, same as the dedicated eraser tool) and a distinct red tint (`#ff5252`) for line/rectangle/ellipse's live drag preview — a raw `fill={null}` preview would otherwise silently render as solid black instead of signaling "this will erase."
- `onContextMenu` is suppressed on the SVG root so a right-click drag doesn't also pop the browser's native context menu.
- Eraser and eyedropper/marquee-select are unaffected — not requested, and eraser already always erases regardless of button.
- Verified with real `PointerEvent`s (`button: 0` vs `2`) in the browser preview: paint-then-right-click-erase round-trips back to an empty canvas for all five tools, the red preview tint renders mid-drag, and `contextmenu`'s `defaultPrevented` confirmed true.

## Ellipse Bug: Only Updating Every Other Pixel While Dragging

User-reported while trying out the new right-click erase on the ellipse tool: dragging from a corner toward the opposite corner only seemed to grow the shape every other pixel, making an exact 16×16 circle unreachable. Root cause: `ellipse.js` derived a center + radius from the drag corners as `(x0+x1)/2` and `|x1-x0|/2`, and `rasterizeEllipse` rounded each independently — for an even-width bounding box that's a half-integer, so two adjacent drag positions (e.g. column 1 vs. column 2) rounded to the *identical* `cx`/`rx` and produced the exact same shape. Confirmed by hand-tracing the rounding sequence for x1 = 0..15 before touching any code.

**Fix:** replaced the midpoint center/radius algorithm with Zingl's ellipse-in-rectangle algorithm, which rasterizes directly from the two corner coordinates — the same calling convention `rasterizeRect` already uses — with no center/radius rounding step to lose resolution at. Prototyped and visually verified via ASCII-rendered grids in a scratch script before touching the real source, to confirm the algorithm was right independent of the existing test suite.

- `rasterizeEllipse(grid, x0, y0, x1, y1, options)` replaces the old `(grid, cx, cy, rx, ry, options)` signature; `ellipse.js`'s `computeEllipseCells` simplifies to forwarding the drag corners directly instead of computing a center/radius from them.
- Existing tests ported to the new corner-based signature; added a regression test asserting every incremental drag step from 1 to 15 in a 16×16 box reaches its own exact edge (false for every even step under the old code), plus a direct check that a 16×16 box rasterizes edge-to-edge as a connected circle.
- Verified in the browser: dragging corner-to-corner across a 16×16 canvas produces a full, unbroken circle touching all four edges, in both outline and filled mode (screenshots taken of both).
