## Work Completed

### Checkpoint A — Canvas.js + store.js (`423458d`)

`Canvas.js` gained `addGrid`/`removeGrid`/`reorderGrid`/`duplicateGrid`
(mirroring the existing layer-level function conventions) and re-scoped
the raw `mergeGridDown` re-export from `Grid.js` into a canvas-level
`(canvas, layerId, gridId)` wrapper; `currentFrameIndex`/`refreshActiveGrid`
were exported (no logic change) so `store.js` could reuse them. `store.js`
gained matching thin-wrapper actions plus `setActiveGridId`; renamed
`updateLayerStyle` -> `updateGridStyle` and `applyPaletteEntryToActiveLayer`
-> `applyPaletteEntryToActiveGrid` (with `PalettePanel.jsx`'s three call
sites updated in lockstep — a required follow-on the original session-15
scope estimate hadn't listed); deleted the dead `setLayerOffset` action;
fixed `setActiveLayerId`/`selectTopLayerAt` to re-resolve `activeGridId`
after a layer switch (previously left stale); and removed
`applyContentSnapshot`'s stale `simpleTier.colorToLayerId` write, which
was silently breaking undo/redo (`canvas.simpleTier` no longer exists on
the model at all) and was the actual reason 4 tests were skipped since
session 1. Test suite: 313 passing / 44 skipped -> 317 passing / 40
skipped / 0 failing — 4 newly-passing from the `simpleTier` fix, 3 more
rewritten against `Grid` instead of asserting on the retired `layer.style`.

### Checkpoint B — LayersPanel.jsx UI, several refinement rounds

Initial build added an expand caret per layer row revealing that layer's
Shapes as sub-rows, each with its own toolbar mirroring `LayersToolbar` one
level down. The user then asked whether that second toolbar could be
dropped in favor of the existing Layers toolbar changing behavior based on
what's selected — after checking convention across Photoshop/Illustrator/
Figma/Affinity/Aseprite (none of them show a second nested toolbar; all
just retarget one shared control), agreed and rebuilt it as a single
context-sensitive toolbar. The tricky part: `canvas.activeGridId`
auto-resolves to a shape via `resolveActiveGrid` even on a plain layer
click (existing, deliberate Session-1 behavior for painting), which would
make layer-level actions unreachable on any non-empty layer if the toolbar
mode were derived from `activeGridId` directly — solved with a separate
`selectionKind` local state, set explicitly by which kind of row was last
clicked, independent of the paint-targeting field.

Several more rounds of user-driven visual polish followed: dropped a
"Shape: {name}" toolbar heading after confirming no editor worth copying
repeats the selected name in a toolbar; gave Add Layer/Add Shape distinct
icons (`AddLayerIcon`/`AddShapeIcon`, a stacked-diamond glyph and a
single-square glyph, each with a small "+" badge) and moved both into the
top toolbar; doubled the thumbnail size to match the two-row name/controls
block, added an "Alpha:" label (later replaced with a half-filled-circle
icon) right-aligned before the opacity input; shrank the expand caret;
deepened the shape indent then eased it back ~25%; and added a small gap
between a layer's row and its shapes so two simultaneously-active
highlight borders don't visually merge.

Two real bugs surfaced during this polish, both fixed same-session:
- An `aspect-ratio` + flex-`stretch` attempt at auto-sizing the thumbnail
  to match its sibling's height blew up to fill available space (an SVG
  has no intrinsic size of its own to stretch from) — reverted to an
  explicit pixel constant.
- Switching the shape row's indent from `paddingLeft` to `marginLeft`
  (to keep the highlight box itself indented, not just its content) while
  still using a `padding` shorthand elsewhere in the same style object hit
  a documented React footgun: removing a longhand that was previously set
  while a shorthand for the same property remains zeroed the removed side
  silently. Fixed by replacing the shorthand with four explicit longhands
  throughout, not just papering over the symptom.

### Checkpoint C — LayerStylePanel + SvgPixelEditor (`f958bee`, `c53c7a2`)

`LayerStylePanel.jsx` retargeted from `canvas.activeLayerId`'s layer to
that layer's active-frame grid matching `canvas.activeGridId`, with the
wireframe's empty-state copy when nothing's selected.
`FillEditor`/`StrokeEditor`/`EffectsEditor` needed zero internal changes —
they only ever read `.style`/`.id` off whatever's passed as `layer` and
call the passed update function generically, so passing a Grid plus a
bound `updateGridStyle` closure satisfied their existing prop contract.

`SvgPixelEditor.jsx` had a live bug, not just a stale convention: both
`clientToCell` and the `GridOverlay` prop derivation read
`activeLayer?.offset.x/.y` — `Layer.offset` doesn't exist on the model at
all since Session 1, so this threw (optional chaining stops at
`activeL?.`, not before `.offset.x`) whenever an advanced-tier layer was
active. Painting had been broken since before this session started; it
only started working again once this fix landed, right at the end.
Both call sites now resolve the active layer's active-frame Grid (via the
newly-exported `currentFrameIndex`) and read `offsetX`/`offsetY` off that.

### Post-verification fix (`2c99029`)

The user noticed "+ Add Shape" started with a pixel already filled in —
`createShapeGrid` always hardcoded a filled 1x1 pixel, correct for
`paintCell`/`paintSimpleCell`'s "first paint allocates a shape" case (that
cell really is being painted) but wrong for the explicit add action, which
`docs/data-model.md` specifies as creating "a new **empty** Grid." Added
an optional `filled` param (default `true`, so existing callers and
`Grid.test.js`'s assertions are unchanged) and had `addGrid` pass
`filled: false`.

## Testing

`npm test` after every checkpoint (317 passing / 40 skipped / 0 failing
throughout — LayersPanel/LayerStylePanel/SvgPixelEditor aren't under
`node --test`, so these numbers didn't move past Checkpoint A, as
expected). Real browser verification via the `pixelyph` dev-server preview

and thumbnail, and applying a palette color to the active shape.

One tooling snag worth recording: the `pixelyph-preview` launch config
runs `npm run preview` (Vite's static preview of the last production
*build*), not the dev server — pointed at it once by habit and got a
confusingly stale UI ("looks like an old version of the application")
before switching to the `pixelyph` config (`npm run dev`, port 5174,
actual live source). Also, `preview_console_logs` stopped returning fresh
entries partway through the session (a frozen/cached buffer, confirmed by
re-checking after several reloads and interactions with an unchanged
entry count) — direct DOM/computed-style checks and screenshots were the
reliable verification signal for the rest of the session, not that tool.

## Notes

- Splitting a large pre-scoped session into checkpoints with an explicit
  pause after each (rather than committing to "finish it all" or "just
  start and see") worked well against a real token-budget constraint —
  each checkpoint left a clean, tested, committed stopping point, and the
  budget question ("do I have enough left for the last piece?") was
  answerable in concrete terms (what shipped smoothly vs. what surfaced
  extra work) rather than a guess.
- The `selectionKind`-vs-`activeGridId` distinction (Checkpoint B) is the
  session's best example of catching a design conflict before writing
  code: deriving toolbar mode from the paint-targeting field directly
  would have quietly broken "move/delete a whole layer" for any non-empty
  layer, since that field's whole job is to auto-resolve to *some* shape.
  Worth remembering as a general pattern — a UI-affordance concern and a
  model's own auto-resolution/stickiness behavior can look like the same
  piece of state without actually being safe to conflate.
- Two bugs this session (the CSS aspect-ratio blowup, the padding
  shorthand/longhand conflict) were caught immediately by the user's own
  visual inspection of the live browser preview rather than by tests or
  static review — a concrete point in favor of this project's browser-
  verification-required policy for anything UI-observable.
