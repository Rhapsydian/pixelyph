## Goal

`BACKLOG.md` explicitly flagged this as the next `/dev-session`, marked to
kick off straight into Session 0's design pass rather than asking what to
work on — the scope was already agreed across session 13's discussion.
Session 0's own deliverable, per that backlog entry, was "a written spec,

## Work Completed

**Research.** Three parallel Explore agents surveyed the current codebase
before any design work started: the model layer (`Layer.js`, `Canvas.js`,
`autoLayerSync.js`, `selection.js`, `history.js`, `GlyphSet.js`, `Grid.js`),
the UI layer (`LayersPanel.jsx`, `FrameStrip.jsx`, `GridOverlay.jsx`,
`SvgPixelEditor.jsx`, the tier toggle, `store.js`), and the save format plus
export consumers (`projectFile.js`, all five export modules). A Plan agent
then drafted an initial spec from that research.

**Design decisions.** The user reviewed the initial draft and corrected or
extended it substantially across several rounds:

- **Literal nesting, not transposed.** The draft proposed a canvas-level
  `frames` array (with grids keyed by layer id) as "cleaner." The user
  pushed back, restating the originally-agreed `Layer[] -> Frame[] ->
  Grid[]` nesting directly. This also turned out to match how
  `Canvas.js` already implements `addFrame`/`duplicateFrame`/`removeFrame`
  today (splice each layer's own `frames` array in lockstep), so it's
  less of a structural change than the transposed version would have
  been.
- **Multi-grid ships now, not deferred.** The draft scoped Advanced tier
  to exactly one Grid per layer per frame, deferring "more than one shape
  per layer" to the separately-parked "Layer groups/folders" backlog item.
  The user rejected that deferral outright: multiple grids per layer needs
  to work from the start. This pulls most of "Layer groups/folders" into
  this migration — that backlog entry was rewritten to reflect it's
  substantially resolved, with only a much narrower residual ask (nested
  *groups of layers*, not multi-shape layers) left open.
- **A real, stored active-grid pointer.** The draft's first pass
  recommended deriving "which grid is active" on the fly with no dedicated
  UI, reasoning that Advanced tier would only ever have 0-or-1 grid per
  layer per frame. Once multi-grid shipped now (previous point), that
  reasoning no longer held. The user wanted an explicit, Illustrator-style
  selectable-shape model instead: `canvas.activeGridId` as real state, with
  a defined resolution algorithm for what happens on a frame switch (try
  the same `id` first — survives `duplicateFrame` for free — then a
  style-match heuristic, then fall back to the first grid in the list).
  Asked directly which approach serves animators best; landed on keeping
  the style-matching fallback specifically because losing selection on
  every frame step during rapid scrubbing is worse friction than an
  occasional wrong-shape guess.
- **Grid gets full control parity with Layer.** Visibility, lock (and
  lockable independent of visibility), opacity, name, thumbnail — the
  same row structure as a Layer row, one level down, per the user's
  explicit "same interface structure" instruction. This also meant adding
  a `Merge Down` action to the per-layer shape toolbar to match
  `LayersToolbar`'s full action set.
- **"Shape" in the UI, "Grid" in code.** The user asked directly which
  word the interface should use. Landed on a split: the app already uses
  "grid" for something else (the pixel-snapping grid overlay toggle), so
  every UI-facing string says "Shape" while the model/type name stays
  `Grid` internally, matching the existing `Grid.js` primitive. Given its
  own explicit section in the spec after the user asked for it to be
  called out rather than left as an inline aside.
- **Two distinct merge operations.** Read the real `mergeLayerDown` in
  `Canvas.js` rather than assume, to check the user's question about
  whether merging preserves today's behavior. Landed on: merging two
  *layers* is now pure concatenation of their grids arrays, per frame — no
  data loss, since each Grid already carries its own style (a real
  improvement over today, which silently discards the top layer's style).
  But merging two *grids/shapes within one layer* necessarily still uses
  the old bounding-box + pixel-OR approach and keeps the *lower* shape's
  style, since a single Grid can only have one style — confirmed directly
  against the user's question, not assumed.
- **Branch the whole migration.** The user asked whether this multi-session
  effort should live on its own branch given the live demo. Checked
  `.github/workflows/deploy-pages.yml` directly — it redeploys on every push
  to `main` — confirming the concern was real: the app will be genuinely
  broken partway through Sessions 1-4 (model shape changes before
  export/UI catch up). Created `layer-frame-grid-redesign`; every commit
  for this multi-session effort lands there until Session 4 (tests) is
  clean, then merges.
- **A standalone data-model doc, not embedded in `BACKLOG.md`.** The user
  wanted the model thoroughly documented with a README callout, separate
  from the session-planning notes. Created `docs/data-model.md`; `BACKLOG.md`
  now points at it rather than duplicating it inline.
- **RLE encoding: considered, deferred.** The user asked to fold in the
  open tokenote item about run-length-encoding grid pixels, since the save
  format is already bumping to v3. Weighed it directly: cropped Grids are
  typically small already, reducing RLE's main benefit, so it stayed a
  separate future item rather than bundling a second pixel-encoding change
  into an already-large migration. Tokenote item left open, not marked
  resolved.
- **Glyph mode follow-up.** After the plan was approved and committed, the
  user asked whether Glyph mode reduces to one grid per glyph. Read
  `GlyphSet.js` directly to confirm rather than answer from memory: yes
  (`glyphToCanvas` wraps a glyph as a single-color Simple-tier canvas, so
  its one layer's one frame can only ever hold 0-or-1 Grid), but
  `canvasToGlyphPixels` needs a real rewrite — it currently reads a dense,
  already canvas-sized buffer, which won't hold once Grids are
  auto-cropped to their bounding box. Added a dedicated "Glyph mode"
  section to the spec and folded `GlyphSet.js` into Session 1's scope in
  `BACKLOG.md` (it wasn't in the original file survey, since it doesn't
  touch `Layer`/`Frame`/`Grid` directly).

**Deliverable.** `docs/data-model.md` (new): terminology section, exact
`Layer`/`Frame`/`Grid` shapes, the active-grid pointer and its resolution
algorithm, grow/shrink-to-fit, both merge operations, UI wireframes, the
save-file v3 migration outline, the Glyph mode note, and a critical-files
list. `README.md` got a new "Data model" section linking to it, plus an
updated Status paragraph. `BACKLOG.md`'s redesign entry now points at the
doc instead of embedding it, is marked Session 0 done / Session 1 next, and
the "Layer groups/folders" entry is marked substantially resolved.

## Notes

- This session is a good example of plan review actually changing the
  design, not just rubber-stamping an agent's draft: three separate points
  (nesting direction, multi-grid scoping, and the active-grid pointer's
  whole approach) were reversed or substantially revised after direct user
  pushback, each time by re-reading real source rather than re-guessing.
- The Glyph mode question came in as a follow-up after the plan was
  already approved and committed — worth remembering that "done" for a
  design-spec session doesn't mean the design can't still have gaps the
  user will spot; re-reading `GlyphSet.js` directly rather than relying on
  the original Explore agent's summary (which had noted it only as
  "tangential — glyphs are single-frame, low risk") is what caught that
  `canvasToGlyphPixels` needs real Session 1 work, not just a passive
  pass-through.
