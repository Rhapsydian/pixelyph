## Phase 5: Font Compilation

**Key decisions and deviations from the plan, discovered during implementation:**

- **No real TTF export exists.** `opentype.js` only ever produces CFF-flavored OpenType (`OTTO` signature) when building a font from scratch — verified empirically (build a font, re-parse it, `outlinesFormat` reports `'cff'`). There's no code path in the library for writing a TrueType `glyf` table for a programmatically constructed font. User confirmed: OTF only, no fake `.ttf` extension on the same bytes.
- **The winding-rule fix the plan called for turned out unnecessary.** Both CFF and TrueType glyph outlines fill under the nonzero winding rule, which only cares about nested contours winding in *opposite* directions from each other, not which absolute direction is "outer." A uniform y-flip (grid space → font space) preserves that opposition, so no per-contour reversal is needed — verified directly with a ray-casting winding-number test against a ring-shaped glyph, not just asserted.
- **`opentype.js`'s CJS and real ESM builds expose different shapes** (default export vs. named-only), which broke the app entirely in the browser (`does not provide an export named 'default'`) despite passing under `node --test`. Fixed with a namespace-import-plus-fallback helper (`opentypeCompat.js`) reused by `ttf2woff`/`wawoff2` too (`woff.js`).
- **`wawoff2` (WOFF2) hangs indefinitely in a real Chromium/Electron environment** — found only by actually driving the app in the browser preview, not by unit tests (it works fine under `node --test`). Reproduced in both `vite dev` and a production `vite build`, so not a dev-server pre-bundling artifact. Root cause wasn't pinned down (plain `WebAssembly.instantiate` works fine in that same environment). Mitigated with an 8s timeout in `toWoff2`.

**Files added:** `pathToContours.js`, `gridToGlyphPath.js`, `compileFont.js`, `opentypeCompat.js`, `woff.js`, `iconFontCss.js`, `demoHtml.js`, `FontExportPanel.jsx`, `polyfills.js` (Buffer shim for `ttf2woff` in the browser).

**Glyph mode selection/copy-paste** (deferred from Phase 3): the store's selection actions (`liftSelection`, `dropFloatingSelection`, `copySelection`, `cutSelection`, `pasteClipboard`, `selectAll`, `cancelFloatingSelection`) are now mode-aware, reading/writing whichever document (`canvas` or `glyphCanvas`) is active — mirroring how `paintCellLive`/`colorAt`/`undo`/`redo` were already made mode-aware in Phase 4. Cross-glyph copy-paste (select in glyph A, switch to glyph B, paste) falls out for free since `clipboard` is a single app-level slot independent of the active document; added a dedicated store-level test for exactly that scenario.

## Testing and Follow-up Round

Manual browser testing (via the preview tool, in real Electron/Chromium — confirmed via `navigator.userAgent`, not a synthetic headless environment) surfaced several real issues `node --test` couldn't catch on its own:

**Zip bundling** (`d3e8e13`): checking multiple export formats triggered one save dialog/download per file. Added a small dependency-free ZIP writer (`export/zip.js`, STORE method only — deliberately not another WASM dependency after the wawoff2 scare) and wired `exportFont` to bundle more than one selected file into a single `.zip`; a single format still saves directly.

**WOFF2 hidden, `BACKLOG.md` created** (`9f7171b`): since WOFF2 reliably times out rather than compiling, left it reachable would mean every export that includes a demo HTML eats an 8s timeout for nothing. Disabled it at the source (`WOFF2_EXPORT_ENABLED = false` in `exportFont`, not just the checkbox) and created `BACKLOG.md` as the shared home for "built, then deliberately hidden pending a fix" items — backfilled the earlier layer-offset UI hide (`1851d64`) into it too, at the user's request, for consistency.

**Icon-font-specific testing found a real bug** (`8df22b6`): asked specifically to test icon-font export (prior testing had focused on character fonts). Creating an icon set, drawing two icons, and exporting with only OTF + CSS/manifest checked (no WOFF) produced CSS whose `@font-face src` referenced `{name}.woff2` and `{name}.woff` — neither of which existed in the export — while never mentioning the `.otf` that *did* exist. `generateIconFontCss` now takes a `formats` option and only references formats actually being exported (priority woff2 > woff > otf, `format('opentype')` for OTF); added a UI guard disabling Export with a warning if CSS/manifest is checked without any binary format alongside it.

**Pixelyph branding footer** (`8b5f971`): every demo HTML now carries a small "Made with Pixelyph" footer linking to the GitHub Pages demo (`PIXELYPH_URL`, the one place to update if an official domain ever exists). Updated the plan file's Phase 8 row to explicitly cover restyling this demo output during the visual design pass, rather than leaving it as a permanent placeholder.

**Documented, not changed:** confirmed and wrote up (comment-only) why the demo HTML always embeds WOFF regardless of the OTF/WOFF checkboxes — those checkboxes only gate the standalone files; WOFF is a lossless, smaller repackaging of the identical compiled font, so there's no rendering-fidelity reason to prefer embedding OTF instead.

**AI-agent leverage — flagged for future discussion, not scoped** (`9b16f63`): discussed the hypothetical of exposing `src/model`/`src/export`'s already-pure functions through a scriptable interface (CLI or MCP server) so an agent could drive Pixelyph via structured calls instead of GUI automation. Added to `BACKLOG.md` explicitly as an unscoped exploration topic, not a planned feature.

## Notes

- Testing methodology this session leaned heavily on driving the actual app in the browser preview (creating projects, painting glyphs via synthetic pointer events, patching `URL.createObjectURL` to capture and structurally parse exported blobs) rather than trusting unit tests alone — this is what caught both the wawoff2 hang and the icon-font CSS bug, neither of which `node --test` could have found.

- `preview_screenshot` intermittently timed out mid-session (`document.hasFocus()` returned `false`, suggesting a stray native dialog or OS-level focus loss); worked around by rendering exported HTML inline via an `<iframe srcdoc>` instead of navigating to a `blob:` URL, and by restarting the preview server.
