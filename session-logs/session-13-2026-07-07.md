## Work Completed

**Symmetry in Glyph mode** (`045497c`) — `canvas.symmetryMode` already
defaulted to `'none'` via `createCanvas`, so `glyphCanvas` (built through
`glyphToCanvas` → `createCanvas`) already carried the field; the actual gap
was narrower. `ContextBar.jsx` only ever read/wrote `s.canvas.symmetryMode`
(the Draw-mode canvas), and `paintCellLive`'s glyph branch
(`store.js`) painted straight into `glyphCanvas` with no `mirrorPoints`
call at all — plus the whole `<select>` was hidden behind an
`!isGlyphMode` guard. Made `paintCellLive`'s glyph branch mirror-aware the
same way the draw branch already is, made `setSymmetryMode` mode-aware
(mutates `glyphCanvas.symmetryMode` and replaces that reference in glyph
mode, `canvas.symmetryMode` otherwise), and made `ContextBar` read/render
based on mode (hidden until an active glyph exists, matching the
`GlyphSizeControl` convention). Verified live: created a Glyph project,
selected Mirror X, and confirmed a single click painted a real mirrored
pair (`M5 8H7V9H5Z` in the resulting SVG path — two adjacent cells, not
one) rather than just the UI appearing.

**Filled toggle in Glyph mode** (`3fedf67`) — purely a UI guard. `shapeFilled`
is a single global toggle, and `rectangle.js`/`ellipse.js` already pass
`ctx.shapeFilled` straight into the mode-agnostic `rasterizeRect`/
`rasterizeEllipse`. Dropped the `!isGlyphMode &&` from the checkbox's render
condition; no other files needed changes.

**Save extension not enforced** (`01400a4`) — every `saveFile(...)` call
site already passes a filename with the correct extension; the bug is that
once a save dialog lets the user retype the filename, nothing re-enforces
it before writing. Electron's `dialog.showSaveDialog` only auto-appends the
filtered extension if the user leaves that filter selected (picking "All
Files," always offered alongside it, drops it) — now the IPC handler
appends the extension back on if the written path is missing it, generic
across every export type the handler backs (zips, SVGs, fonts, not just
`.pixelyph`). The web build's `showSaveFilePicker` call was only passing
`suggestedName`, no `types` filter — added one, which is what makes Chrome
auto-append the extension the same way a native dialog does. The plain
`<a download>` fallback (old Firefox/Safari only) has no post-write hook to
correct this and was left as a documented, accepted gap.

**Pixel-format review → bit-packing** — this is the item that took the
session in an unplanned but productive direction (see Notes). Landed on:
a grid cell in `frame.pixels` is always exactly 0 or 1 (color/style lives
on the Layer, confirmed directly in `Canvas.js`'s `paintCell`/
`paintSimpleCell`), so spending a full byte per cell before base64 wastes
7 of 8 bits. Added `bitsToBase64`/`base64ToBits` to `projectFile.js`,
bumped `PIXELYPH_VERSION` to 2, and threaded the document's version into
`deserializeLayer`/`deserializeGlyph` so pre-v2 saves still decode via the
old one-byte-per-cell path. Only `projectFile.js` changed — `Canvas.js`,
`Grid.js`, and all in-memory paint code still work with plain 0/1
`Uint8Array`s throughout; this is purely a serialize/deserialize-boundary
change (`c33e016`).

## The pixel-format review became a design discussion

The user tested the bit-packing claim themselves — a 16×16 single-layer
save producing a 344-character `pixels` string — which checked out exactly
(256 raw bytes → `⌈256/3⌉×4` = 344 base64 chars) but led to a sharper
question: since a layer's style already lives at the layer level, why does
each cell cost a full byte instead of one bit? That's a real, valid
optimization (implemented above), but then the user asked a bigger
question: **how does the data model handle a single new color painted on
just one frame of a 16-frame animation?**

Traced this precisely through `autoLayerSync.js`/`Canvas.js`: a brand-new
auto-managed `Layer` gets created with a full `frames` array (one entry
per `frameCount`, all-zero `Uint8Array`s) — only the actively-painted
frame's cel gets real content, and the layer is never pruned once any
frame has content, so 15 empty cels ride along for the animation's entire
life. The user proposed inverting the model to `Frame -> Layer[]` to fix
this. Before agreeing, scoped the actual blast radius with a read-only
Explore agent: ~17 files, 12 needing real algorithmic rewrites (not
find-replace) — `Canvas.js`'s frame-management functions, `composeLayersSvg.js`'s
per-frame compositing, three animated-export files — plus ~7 test files,
estimated at 2-4 days.

That number was big enough to research first rather than commit to
blindly: a web search on how Adobe Animate/Flash, Aseprite, After
Effects/Lottie, and Rive actually model layers vs. frames found that
**none of them invert the nesting** — every one keeps Layer as the
persistent identity and solves the exact sparsity problem by making
per-frame content sparse/optional (Animate's keyframe-then-hold; Aseprite's
optional per-frame cels) rather than restructuring the hierarchy. Presented
this back to the user, who then reframed the actual problem: Pixelyph has
been conflating "layer" (a persistent, styled identity) with "shape" (the
actual pixel content) — closer to how a vector editor's "layer" is really
a collection of independently-styled objects. Landed, across several
rounds, on a confirmed design:

```
Canvas -> Layer[] (identity: name, locked, opacity, z-order, no style)
       -> Frame[] (same count/order across every layer, as today)
       -> Grid[]  (independent per frame; each: offsetX, offsetY, width, height, pixels, style)
```

Key resolved decisions: Grids have no cross-frame identity (each frame's
`grids` list is independent, like today's per-frame pixel copies); Simple
tier collapses to exactly one `Layer`, ever, with per-color Grids inside
each frame (simpler than today's per-color-`Layer` scheme, not more
complex); there's always an "active grid" being edited, and Simple tier's
paint op turns a cell off in whichever grid currently owns it and on in
the target color's grid (same mutual-exclusivity invariant
`paintSimpleCell` already enforces, re-scoped from sibling layers to
sibling grids); Grid bounds are fully auto-computed, both growing (reusing
`Layer.js`'s existing `growToInclude` almost as-is) and shrinking (new
logic, plus pruning an emptied Grid entirely); z-order is plain array
order, same as today's layers.

Two nice side effects surfaced during this: the auto-computed-offset
decision directly retires the parked "Advanced-tier layer offset — manual
X/Y input hidden" backlog item (that item was blocked on fractional
manual-offset values disagreeing with the grid-overlay/cursor-snap
pipeline; if offset is never manually typed, that bug class can't occur),
and "a Layer holding more than one Grid" is most of the way to the parked
"Layer groups/folders" item already.
