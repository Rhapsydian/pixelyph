## Work Completed

### Checkpoint 1 — Tile preview row/column gaps (`2a9fe28`)

`TilePreviewPanel.jsx`'s grid container set `gridTemplateColumns` but never
`gridTemplateRows`, so rows fell back to `auto` sizing — which picked up
the injected `<svg>` markup's inline-baseline gap (the exported SVG has no
`width`/`height`/`display` of its own), while columns' explicit `1fr`
tracks were separately rounding-sensitive at certain widths. Fix: added
the matching `gridTemplateRows`, plus a `.tile-preview-cell` CSS rule
forcing the injected SVG to `display: block` and fill its cell.
Browser-verified at 3×3 and 6×6, both backgrounds, two viewport widths —
seamless in all cases, confirmed via computed-style inspection
(`grid-template-rows` now matches `grid-template-columns` exactly).

### Checkpoint 2 — Context menu after off-canvas right-drag (`b9920af`)

`SvgPixelEditor.jsx`'s `setPointerCapture` redirects `pointermove`/
`pointerup` back to the `<svg>` regardless of cursor position, but has no
effect on the native `contextmenu` event, which hit-tests wherever the
cursor actually is — so releasing a right-drag outside the canvas let the
browser's context menu open on whatever was underneath, since the `<svg>`'s
own `onContextMenu` only helped when the event targeted it directly. Fix:
a `rightDragRef`, gated document-level `contextmenu` listener mounted for
the component's whole lifetime (not added/removed per-gesture, since
`contextmenu` fires synchronously right after `pointerup` for the same
button release — a listener torn down inside `handlePointerUp` could lose
that race). Verified via synthetic pointer/contextmenu event dispatch
(stubbing `setPointerCapture`, which rejects synthetic pointer ids with no
real active pointer): a contextmenu dispatched on `document.body` after a
simulated off-canvas release was correctly prevented, the ref cleared
afterward so an unrelated right-click still shows the normal menu, and
on-canvas erase/context-menu-suppression behavior was unaffected.

### Checkpoint 3 — Layer click stealing shape selection (`a43d5a8`)

`Canvas.js`'s `refreshActiveGrid` always resolved with `prevGrid: null`, so
re-deriving `activeGridId` after *any* layer-panel click reset to the
layer's first shape — even when the layer didn't actually change,
overriding a shape the user had explicitly selected via a `ShapeRow`
click. Fix: `refreshActiveGrid` takes an optional `prevLayerId`; only when
it's non-null and equals the (already-mutated) `activeLayerId` — i.e. no
real layer change happened — does it pass the currently-active grid
through as `prevGrid`, so `resolveActiveGrid`'s id-match branch keeps it
selected. Threaded through `store.js`'s `setActiveLayerId` and the
eyedropper's `selectTopLayerAt` (same root cause, same fix, not separately
reported). Browser-verified both directions: re-clicking the already-active
layer preserved an explicitly-selected non-first shape; switching to a
different layer and back correctly reset to that layer's first shape (no
stale carryover).

### Checkpoint 4 — "Active shape" selection scope, default (`d47768d`)

`ContextBar.jsx`'s "Select from:" dropdown only offered "Active layer"/
"All visible layers"; there was no way to scope Select/Copy/Cut to just
the one shape currently selected in the Layers panel. Added
`extractRectFromActiveGrid` (`selection.js`), reusing `Grid.js`'s `get()`
the same way `paintCell`/`topGridAt` already do rather than reimplementing
cell-ownership logic, wired into `store.js`'s scope routing as a new
`'activeShape'` value and made it the default (previously `'activeLayer'`).
An interesting non-bug found along the way: `clearRect` already only ever
erases the active *grid*, not the whole active layer — so the existing
`activeLayer` scope had a pre-existing read/clear asymmetry (read scans
the whole layer, clear only touches the active shape); the new
`activeShape` scope needed no new clear logic at all, since `clearRect`'s
existing behavior was already exactly right for it. Flagged, not fixed —
outside this session's four items. Browser-verified: dropdown defaults to
"Active shape"; a marquee spanning two shapes in the same layer correctly
excludes the non-active shape's pixel under "Active shape" scope (1 cell
copied) and correctly includes both under "Active layer" scope (2 cells).

## Notes

- **The tier redesign, in brief:** discussion surfaced that "Simple vs.
  Advanced" today bundles two independent things — "how many layers can
  exist" and "how much manual shape/style control you get" — into one
  flag. The agreed direction: every tier gets real multi-layer support
  (each layer auto-managing its own per-color shapes in the new "Pixel"
  tier); the only thing that stays tier-gated is manual shape/style
  authoring. Since `canvas.tier`'s stored values (`'simple'`/`'advanced'`)
  don't need to change — only the *displayed* labels do — the rename to
  "Pixel"/"Shape" carries zero save-format risk, which is what made it
  reasonable to fold into the same plan as the layers work rather than
  treating them as separate asks.
- A Plan agent's first draft of that plan had two things worth catching
  before approval: it drafted a confirm-dialog string with an unescaped
  apostrophe inside a single-quoted JS literal (would have broken the
  file), and its ASCII wireframe didn't match the Layers panel's actual
  current two-line row layout (name on its own line, then eye/lock/opacity
  below it) — verified and fixed both against the real UI before writing
  the final plan.
- Wasted some time mid-session verifying Checkpoint 1 against the wrong
  preview server (`pixelyph-preview`, which runs `vite preview` against
  the stale `dist/` build) before realizing the CSS class wasn't showing
  up in the DOM because the server wasn't picking up source edits at all
  — switching to the actual `pixelyph` dev-server config fixed it
  immediately. Worth double-checking which `launch.json` entry is the real
  dev server vs. a production-preview server before debugging "why isn't
  my change showing up."
- Verifying Checkpoint 2's off-canvas drag fix needed synthetic
  `PointerEvent` dispatch (no real mouse to drive), which hit a real
  browser restriction: `setPointerCapture` throws on a synthetic pointer id
  with no genuine active pointer. Temporarily stubbing
  `Element.prototype.setPointerCapture` for just the dispatch made the
  rest of the gesture handling testable without weakening the actual fix
  being verified.
