### Checkpoint 5 — external-paste color-handling toggle (`98250a5`)

New `buildGridCloneUnioned` (`selection.js`, sibling to `buildGridClonesByColor`)
unions every non-empty pasted pixel into one Grid clone painted with the
active color, instead of one clone per distinct color. `pasteColorMode`
(`'multiple'`/`'single'`) plus `pasteRaw`/`touched` bookkeeping on
`floatingGridSelection` let a new `ContextBar.jsx` toggle — "Paste as:
Multiple shapes (by color) / Single shape" — regenerate the pending clones
once, right after paste, then lock out once the user moves or transforms
the pending selection. Caught and fixed a real bug during live testing
before committing: `pasteImageBlob` applied `pasteColorMode` even to a
single-color paste, silently recoloring it to the active color whenever
`'single'` was left selected from an earlier paste — fixed by only
branching to the unioned path when there are actually 2+ distinct colors.
Tests: 455/455 → 458/458.

### Bug found live: partial-lift preview hid a shape's un-selected remainder (`84591c4`)

User report: "when the floating selection comes up... the non-selected
areas of the affected shapes disappear from the canvas." Root cause:
`buildFloatingGridPreviewDoc` excluded a destructively-lifted grid
*entirely* from the pending-preview render rather than hiding just the
cells the clone actually came from — so marquee-dragging or
Transform-ing only *part* of a multi-cell shape made the rest of that
same shape vanish from view for the whole pending state, even though the
real document still had it and finalize was always correct. All prior
testing (this session's and session 29's) had only ever exercised
single-cell or fully-selected shapes, so the gap was never exercised.
Fixed: the real grid now renders as a remainder (its own pixels minus the
lifted snapshot's cells) instead of being dropped outright. New
regression test in `selection.test.js`; live-verified with a 4-cell shape,
lifting only part of it.

### Bug found live: undo/redo silently changed the active shape (`f6d2087`)

User report: "undo is changing the selected shape in a weird way." Root
cause: `applyContentSnapshot` (run by every undo/redo) called
`refreshActiveGridModel(canvas)` without the canvas's own `activeLayerId`,
so `refreshActiveGrid`'s "stay on the same shape" check was always
skipped and `resolveActiveGrid` fell back to the active layer's *first*
shape on every single undo/redo, regardless of what was actually
selected — an older bug, unrelated to this session's redesign, just never
noticed until now. Fixed: pass the pre-restore `activeLayerId` through,
matching `setActiveLayerId`/`selectTopLayerAt`'s existing pattern. New
regression test proven to fail on the old code first (verified via
`git stash`), then pass after the fix.

### Bug found live: external image paste upscaled small sources (`166c790`)

User report: pasting a 4×4 raster image scaled up to fill the canvas.
Root cause: `pasteImageBlob` always called `importRasterToGrid(image,
doc.width, doc.height)`, resampling *every* pasted image to the canvas's
own dimensions regardless of the source's actual size — correct for a
screenshot larger than the canvas (downsamples to fit, the feature's
original intent), wrong for a source already smaller than the canvas
(upscaled instead of landing at its own size). Fixed: clamp the target
size to `min(image dimension, doc dimension)` per axis. Live-verified both
directions: a 4×4 source now pastes at native size (centered), a 200×200
source still downsamples to fill a 16×16 canvas, unchanged.

### Bug found live: Move tool ignored "Select from: Active layer" (`842216a`)

User report: the Move tool should hit-test the topmost visible shape at
the clicked cell, then move every shape in that shape's layer, when scope
is Active layer — instead it always moved just the single clicked shape
regardless of scope. Root cause: `targetMoveTool` never read
`selectionScope` at all. Fixed: the existing hit-test is unchanged, but an
`activeLayer` scope now drags the hit shape's whole layer via the already-
generic `nudgeLayerFrameLive` (already used for Pixel tier/Glyph mode's
whole-layer drag, and already correct for multi-grid layers since it
shifts every grid in a frame uniformly) instead of the single-shape
`setGridPropsLive` path. Added `test/ui/draw/tools/targetMove.test.js` —
this tool had no test coverage at all before this session, following the
project's existing mock-`ctx` pattern (`pencil.test.js`).

## Notes

- **`computer` tool coordinates are screenshot-pixel space, not real
  viewport space.** Feeding it coordinates computed from
  `getBoundingClientRect()` (real space) silently lands clicks on the
  wrong cell — this caused a confusing detour early in the checkpoint-6
  live verification (a painted shape landed at an unintended offset).
  Once noticed, switched to dispatching raw `PointerEvent`s with real
  `clientX`/`clientY` via `javascript_tool` for all precise canvas
  interaction for the rest of the session — reliable throughout.
- **The `computer` tool's `left_click` did not reliably fire usable
  pointer events for this app** in a couple of spots this session (a
  plain paint click on a freshly-created empty shape produced no visible
  effect at all) — raw `PointerEvent` dispatch worked every time it was
  tried instead. Consistent with session 29's own notes on this tool's
  sharp edges with this app's pointer-event-driven canvas.
- Clicking inside an *already-selected* marquee rect starts a drag-move
  of that selection rather than a fresh selection — worth remembering
  when scripting a new selection immediately after an old one, since a
  pointerdown landing inside the stale rect silently reinterprets the
  gesture. Escape-cancel first (a true no-op) before starting a new
  selection from scratch avoids this.
- Two of this session's four extra bug fixes (`applyContentSnapshot`'s
  missing `prevLayerId`, `targetMoveTool`'s missing `selectionScope`
  check) were pre-existing bugs unrelated to this session's redesign,
  just surfaced by the user exercising newly-shipped features more
  thoroughly — not regressions introduced by the checkpoints themselves.
