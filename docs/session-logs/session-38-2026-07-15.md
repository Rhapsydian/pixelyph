# Session 38 — 2026-07-15

## Goal

The user had already pointed GitHub Pages at the `pixelyph.com` custom
domain in repo settings; this session's job was to bring the codebase in
line with that — the build was still hardcoded to assume the old
`rhapsydian.github.io/pixelyph` project-page subpath.

## Decisions and rationale

- **`vite.config.js`** — `base` was previously `mode === 'itch' ? './' :
  command === 'build' || isPreview ? '/pixelyph/' : '/'`, i.e. build/preview
  used the `/pixelyph/` subpath while the dev server used root. A custom
  domain serves from the root, so build and dev now agree: `mode === 'itch'
  ? './' : '/'`. This also let the now-unused `command`/`isPreview`
  destructured params drop out entirely — simpler than before, not just
  updated.
- **`public/CNAME`** — new file, contents `pixelyph.com`. Reasoning: this
  repo's Pages workflow (`deploy-pages.yml`) uses
  `upload-pages-artifact`/`deploy-pages` against `dist/` directly, not a
  branch-based publish GitHub can auto-manage a CNAME file on. Vite copies
  `public/` into `dist/` verbatim, so this is the right place to make the
  custom domain persist across every future deploy without touching the
  workflow file.
- **`src/export/font/demoHtml.js`** — `PIXELYPH_URL` (the "Made with
  Pixelyph" footer link on exported demo HTML) had a comment reading
  "update it if Pixelyph ever gets an official domain name" — that
  condition is now true, so it was updated to `https://pixelyph.com/` and
  the now-obsolete comment removed.
- **`test/export/font/demoHtml.test.js`** — updated its assertion to match
  the new URL.
- **`README.md`** — live-demo link repointed at `pixelyph.com`; the
  itch.io mention was also reworded from "also playable on" to "also
  available on" at the user's explicit request (unrelated to the domain
  change itself, just bundled into the same pass); Status paragraph got a
  short new sentence noting the domain migration.

## Problem-solving notes

- When a custom-domain migration is in flight, remember to grep beyond the
  obvious config file — this session found a second hardcoded reference
  (`demoHtml.js`'s `PIXELYPH_URL`) that a narrower "just fix vite.config.js"
  pass would have missed, plus the test asserting on it.
- For a live "is this actually broken or just my browser" question, it's
  faster and more convincing to check the real URL directly (Browser pane:
  load `https://`, then try the bare domain and see whether it upgrades)
  than to reason about it in the abstract — this confirmed the server side
  was fine in under a minute and pointed the user at the right (client-side)
  fix.
