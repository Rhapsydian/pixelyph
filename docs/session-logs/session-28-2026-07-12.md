### Round two — user-reported Shape-tier corruption bugs (uncommitted)

After the three checkpoints shipped, the user reported, against their own
real project (loaded in the Browser pane via "Continue Last Session" —
same origin/IndexedDB as the dev server, not a synthetic test): selecting
an *entire* shape and transforming it corrupted a *different* shape in the
same layer, and Active Layer scope "doesn't work correctly at all." Mid-
investigation the user gave a third, architectural correction: the
transform must operate on each shape's own grid data directly (style must
never be affected), and *every* shape with cells in the selected area must
transform, not just the topmost cell per position.

**Root causes**, all in the pre-existing flat-cell-color machinery
(`liftSelection`/`clearSelectionRect`/`pasteCells`, `selection.js` +
`store.js`) that Checkpoint 2 had built on top of:
- `clearSelectionRect` only special-cased `allVisible` scope; `activeLayer`
  fell through to `clearRect`, which only ever erases
  `canvas.activeGridId`'s one grid — under-clearing when the active layer
  has more than one shape in the rect. Fixed with a new
  `clearRectFromActiveLayer` (mirrors `clearRectAllLayers`, scoped to one
  layer).
- Destructively clearing a *fully*-selected shape empties and deletes its
  Grid, which reassigns `canvas.activeGridId` to a sibling shape
  (`paintCell`/`eraseFromLayer`'s existing "fall back to `frame.grids[0]`"
  behavior) — so the later paste-back silently painted into that sibling.
  Fixed by having `liftSelection` capture `activeGridId` before the
  destructive clear and restore it after.
- `pasteCells` always painted through `canvas.activeGridId` regardless of
  each cell's own color — a multi-color paste (`activeLayer`/`allVisible`
  scope) collapsed every color after the first into one grid/style. Fixed
  by grouping cells by color for advanced tier before painting.

These three fixes resolved the *duplication/deletion* symptoms but not the
user's architectural point: flat-cell extraction fundamentally can't
preserve gradient/stroke/effects (confirmed live — a gradient-filled
shape's overlapping cells extracted as a flat gray placeholder, then got
pasted back as a real, wrong, solid-gray shape).

**Redesign**: for advanced (Shape) tier with nothing already floating,
Transform > Selection now bypasses `liftSelection`/`floatingSelection`
entirely. New `transformGridRegion` (`Grid.js`) flips/rotates only the
portion of one Grid's own pixel buffer inside the selection rect, remapped
around the rect's own bounds, in place — never touches `style`, never
merges shapes. New `transformSelectionRegion` (`Canvas.js`) resolves which
Grids a scope puts in play (`activeShape`: the active grid only;
`activeLayer`: every visible/unlocked grid in the active layer;
`allVisible`: every visible/unlocked grid in every visible layer) and calls
`transformGridRegion` on each independently. `store.js`'s
`transformSelectionInStore` branches to this path first; Pixel tier, Glyph
mode, and the rare case of an already-floating Shape-tier selection still
use the original flat-cell path (no per-shape style to lose there).
Commits immediately — one undo step, no floating-preview stage at all for
Shape tier.

## Notes

- **A destructive selection clear can delete the very shape being
  selected, which has a real side effect worth remembering elsewhere
  too**: `paintCell`/`eraseFromLayer`'s "grid became empty → remove it →
  if it was the active grid, fall back to `frame.grids[0]`" behavior is
  old, correct, load-bearing logic for ordinary erasing — but it means any
  *new* code that destructively clears a whole shape via the normal
  per-cell erase path can't assume `canvas.activeGridId` still points at
  what it did when the clear started. Capture-and-restore around the
  clear, not a read-fresh-each-time assumption.
- **Flat, per-cell color extraction is the wrong abstraction for Shape
  tier**, full stop — not just for the Transform feature. It was already
  the mechanism plain marquee drag-move uses too, and shares the same
  gaps (this session's three targeted fixes benefit that path as well,
  even though it wasn't the reported bug). Any future Shape-tier
  selection feature should default to a per-Grid buffer operation
  (`transformGridRegion`'s pattern) rather than reaching for the
  Pixel-tier-shaped `liftSelection`/`floatingSelection`/`pasteCells` trio.
- **Verify DOM state directly, not just screenshots, once a fix claims to
  touch multiple objects at once** — the undo bug above was invisible in
  screenshots (a subtle stripe-orientation difference) and was only caught
  by diffing the rendered SVG `path` `d`/`transform` attributes before and
  after. Worth doing by default for any multi-shape assertion, not just
  when something already looks suspicious.
- **The Browser pane was unusually unstable this session**: `computer`
  screenshot actions timed out repeatedly mid-session (recovered after
  opening a fresh tab and letting the layout settle), and the canvas SVG's
  measured screen position was observed to alternate between two different
  values across consecutive reads for no apparent reason, which silently
  broke coordinate-based marquee drags until re-measured immediately
  before each drag. Server-side health was confirmed independently via a
  direct `curl` (fast, healthy) throughout — this was tooling/rendering
  flakiness, not an app or server problem. Consistent with session 26/27's
  own notes about Browser-pane quirks (stale console-message history,
  disconnected `import()`'d store instances) — a recurring category worth
  double-checking against before concluding the app itself is broken.
- The user's mid-session architectural correction (grid data, not flat
  colors) arrived *after* three targeted fixes were already implemented
  and tested — those fixes turned out to still be correct and worth
  keeping (they fix the same underlying machinery's use by plain drag-move
  too), just insufficient on their own for Shape tier's Transform feature.
  Worth remembering: a bug-hunting session's first fix attempt being
  superseded by a deeper redesign doesn't mean the first attempt was
  wasted work, if it's independently correct for the narrower case it
  covers.
