## Key decisions and rationale

- **Resumed directly from the backlog, no re-scoping needed**: `BACKLOG.md`
  already had the exact resume checklist from session 22's pause, matching
  this project's established "pre-scoped kickoff" pattern — verified the
  paused `store.js`/`MenuBar.jsx`/`TransformScopeModal.jsx` diff was
  actually sound (via an Explore pass reading the current files, not just
  trusting the backlog note) before treating it as ready to verify-and-ship
  rather than re-plan.
- **Checkpoint E's divider styling reused an existing precedent**: rather
  than inventing new CSS, `.context-bar-divider` was modeled directly on
  the already-shipped `.tool-rail-divider` (`theme.css`), just rotated to a
  vertical 1px rule with `align-self: stretch`.
- **`hasToolSpecificControls` gate added to avoid a dangling divider**: the
  original Checkpoint E design didn't say what happens when the right-hand
  group would be empty (e.g. a tool with no tool-specific options and Pixel
  tier's select-scope hidden) — added a computed boolean so the divider and
  right-hand wrapper only render when there's actually something in them,
  rather than showing an empty divider floating at the far edge.
- **Left-align follow-up**: after Checkpoint E shipped, the user asked to
  left-align the ContextBar's tool-specific group instead of pushing it to
  the bar's far right edge — a one-line fix (drop the group's
  `marginLeft: 'auto'`), verified and committed as its own small change
  rather than folded into the Checkpoint E commit, since it landed after
  that commit was already made.
- **Tier-switch bug: fix belongs in the model, not the store** — traced the
  bug to `convertTier` (`Canvas.js`) rather than `setTier` (`store.js`):
  `paintSimpleCell` (Pixel tier's paint routine) never touches
  `activeGridId` at all, so it's stale by the time Shape tier's `paintCell`
  looks a grid up strictly by id. Rather than patching around this in the
  store, gave `convertTier` an optional `activeColor` parameter and had it
  resolve `activeGridId` to the active layer's active-frame Grid matching
  that color — keeping the tier-conversion logic (like the rest of
  `Canvas.js`) a pure, directly-testable function, with `store.js` only
  responsible for passing `get().activeColor` through. Chose an explicit
  parameter over having `Canvas.js` reach into store state, preserving the
  existing pure-model/store separation the rest of the codebase already
  follows.
- **Fallback behavior on no color match**: rather than leaving
  `activeGridId` unresolved (null) when the active color was never painted
  on that layer, fell back to the existing `resolveActiveGrid` default
  (first shape, or none) — matching prior tier-switch behavior exactly in
  the case this fix doesn't apply to, so nothing regresses for a layer with
  no matching shape.

## Notable problem-solving

- Hit a DOM-read timing gotcha while manually verifying in the Browser
  pane: reading page state synchronously right after `.click()`-ing a
  store-wired button (via `javascript_tool`) sometimes showed the
  *pre-click* state, because React batches the actual re-render after the
  event handler returns rather than flushing it synchronously within the
  same script. Confirmed this was a read-timing artifact, not a real bug,
  by re-running the same click-then-read sequence with a short
  `setTimeout` between them — recorded as a new testing-methodology note
  in `BACKLOG.md` for future sessions using the same verification approach.
- The tier-switch bug's root cause took one targeted Explore pass to
  isolate precisely because Pixel tier's `paintSimpleCell` and Shape
  tier's `paintCell` resolve "which shape to paint into" via two entirely
  different mechanisms (scan-by-color vs. lookup-by-id) that happen to
  share no code path — worth remembering that `activeGridId` is a Shape-
  tier-only concept that Pixel tier's paint routine has no reason to keep
  in sync, by design.
