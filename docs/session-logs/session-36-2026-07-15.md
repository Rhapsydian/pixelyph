## Shape-persistence fix (commit `c5953ce`)

**Root cause:** `shrinkGridToFit` (`src/model/Grid.js`) returns `null`
(without mutating) when a Grid has no pixels left. Three call sites treated
a `null` return as "delete this shape from `frame.grids`" —
`paintCell`/`eraseFromLayer` (`src/model/Canvas.js`) and
`pruneEmptyGridsForSelection` (`src/model/selection.js`, Cut's cleanup
pass). A fourth call site, `autoLayerSync.js`'s `paintSimpleCell` (Pixel
tier's auto-managed per-color grids), was confirmed out of scope — those
grids have no user-facing identity, so auto-delete-on-empty stays correct
there.

**Decision:** rather than change `shrinkGridToFit`'s own contract (which
the out-of-scope Pixel-tier caller still needs), added a new
`collapseToEmptyGrid(grid, anchorX, anchorY)` primitive that collapses a
grid to a persistent 1×1 all-zero shape — the same representation
`createShapeGrid({ filled: false })` already produces for "+ Add Shape".
The three in-scope call sites now call this instead of deleting. Cut's
path was included in scope after explicit confirmation from the user
(originally reported as an erase-only bug, but Cut used the identical
"mirrors eraseFromLayer's auto-delete convention," so leaving it unfixed
would have been visibly inconsistent).

Added/updated tests in `test/model/Canvas.test.js`, `test/state/store.test.js`.
Verified live in-browser (not just `node --test`) by driving the actual
running store/render pipeline via `javascript_tool`, since the Browser
pane's `computer` screenshot action was unreliable this session (see
Problem-solving notes).

## Clipboard Copy/Cut → Paste fixes (commit `b53d6c7`)

Testing the shape-persistence fix surfaced two more, pre-existing bugs in
the same area, both isolated via direct code reading rather than guesswork:

1. **No floating preview after Paste.** `snapshotGridClipboard`
   (`src/state/store.js`) dropped `visible`/`locked`/`name` when freezing a
   floating selection onto the clipboard; `pasteClipboard`'s reconstructed
   clone then had `grid.visible: undefined`, which `composeLayersBody`'s
   `.filter((grid) => grid.visible)` (`src/export/svg/composeLayersSvg.js`)
   silently dropped — while the separate dashed-outline indicator in
   `SvgPixelEditor.jsx` rendered regardless, giving the appearance of an
   empty selection box.
2. **Paste always created a new shape.** `pasteClipboard` hard-coded
   `originGridId: null` on every rebuilt clone, so `finalizeGridSelection`
   (`src/model/selection.js`) never took its existing (already-working, used
   by the in-canvas drag-Move flow) merge-into-real-grid branch.

**Product rules, agreed with the user through a few rounds of
clarification:** internal (this-app) Copy/Cut → Paste merges into a shape
instead of creating a new one, when there's an unambiguous target — a
single-shape source always merges into whichever shape is currently active
at paste time; a multi-shape source (Active-layer-scope Copy/Cut of several
shapes at once) restores each piece into its own original shape only if the
active layer hasn't changed since the copy/cut, otherwise every piece
becomes a new shape in the current layer (matching what external
OS-clipboard image paste already does, which is untouched by any of this).

Implementation added `originGridId`/`originLayerId` tracking to the
clipboard snapshot (previously deliberately stripped to keep the clipboard
"portable"), branching logic in `pasteClipboard`, and a guard in
`clearGridSelectionSource` for the new case where a clone has a real merge
target but no live position to clear from. Added tests across
`test/state/store.test.js` and `test/model/selection.test.js` covering all
four cases (visible preview, single-shape merge, locked-target fallback,
same-layer multi-shape restore, different-layer multi-shape new-shapes).
Verified live via the running store, same technique as above.

## itch.io publishing attempt (commits `c769a54`, `c057ecb`, `fd43ca5`)

Added `build:itch` (`vite build --mode itch`) with a relative `base: './'`
in `vite.config.js`, alongside the existing GitHub-Pages-targeted default
build — confirmed via grep that no Electron-specific code exists in `src/`,
so the same renderer source works unmodified as a plain web build.

While investigating why the itch page showed a blank/gray screen, noticed
`src/export/font/woff.js` statically imported `wawoff2` even though
`WOFF2_EXPORT_ENABLED` (already `false`, see BACKLOG's WOFF2 entry) means
it's never actually invoked — a static import still bundles a dependency
regardless of runtime reachability. Converted to a dynamic
`import('wawoff2')` inside `toWoff2`; the minifier then proved the whole
disabled branch dead and dropped `wawoff2`/`toWoff2` entirely from the
build. Main JS chunk: 2,107.70 kB → 869.08 kB (gzip 768.94 kB → 260.51 kB).

**Blocked:** isolated, via three minimal throwaway test-zip uploads to the
live itch project (not committed to the repo), that itch.io's own HTML5
unpacker 404s on any file inside a subfolder while root-level files serve
fine — reproduced across three separate fresh uploads and on the real
build's `assets/`/`manual/`/`samples/` folders. Not a bug in our build
(verified: relative paths, zip structure, and case-sensitivity are all
correct on our end). A flat-output workaround was fully scoped (in plan
mode) but deferred at the user's request rather than spend further upload
attempts before hearing back from itch support — see BACKLOG.md's new
entry for the exact plan. Drafted a support-ticket writeup for the user
using the 3-test reproduction as evidence.
