## Key decisions and rationale

- **Checkpoint 6 verification method**: rather than hand-deriving offset
  math for every case, verified canvas-level rotate by painting a known
  mark, rotating, and reading the rendered SVG's `translate(...)` back out
  — this caught the exact offset (`translate(14,1)` on a 10×16→16×10
  rotation) matching hand-derived predictions to the pixel. Shape/layer
  cases were verified visually (screenshot before/after) since the
  underlying math was already unit-tested in a prior session; layer-level
  rotate on a non-square canvas was found to push content off-canvas,
  which — after checking `BACKLOG.md`'s existing "off-canvas content is
  preserved, not deleted" stance — was judged working-as-designed, not a
  bug worth fixing.
- **Gradient handle scope (Checkpoint 7 sub-step 1)**: linear only, radial
  deferred — matches the tool-roadmap doc's own recommendation to split
  angle-drag (simple, one rotation handle) from radial cx/cy/r (three
  degrees of freedom) into separate sub-steps.
- **Gradient handle live/commit split**: mirrors the existing
  `paintCellLive`/`commitStroke` pattern exactly — a new
  `updateGridStyleLive` mutates `grid.style` in place with no `commit()`,
  so a whole drag collapses into one undo step regardless of how many
  pointermove events fire.
- **Gradient handle later gated behind a toggle**: after the first
  browser demo, the user asked for this to be tool-driven rather than
  always-on — added a per-shape "Show angle handle on canvas" checkbox in
  the Style tab (defaulting off, never remembered across shapes), visible
  only while the Style tab itself is the active side-panel tab. Required
  lifting `SidePanel.jsx`'s tab-selection state out of local `useState`
  into the store (`sidePanelTab`) so `SvgPixelEditor.jsx` — a sibling
  component — could read which tab is currently showing.
- **New rotation directions (180°/CCW) as repeated 90° passes**: rather
  than writing new pixel-transform math, `rotateCanvas180`/
  `rotateCanvasCCW90`, `rotateActiveGlyph180`/`rotateActiveGlyphCCW90`, and
  later `rotateActiveLayer180`/`rotateActiveLayerCCW90` and
  `rotateActiveShape180`/`rotateActiveShapeCCW90` are all built by looping
  the existing, already-tested 90°-clockwise primitive N times (2 for
  180°, 3 for CCW), confirming/committing only once. For canvas rotation,
  width/height must swap once *per pass* (not once total), since each
  pass's offset math needs the canvas dimensions as they stood after the
  previous pass. For glyph rotation, only the *first* pass in a multi-pass
  sequence can ever need the destructive pixelsPerEm re-crop — verified by
  reading `rotateGlyph90`'s actual body before committing to the approach
  — so the existing confirm-dialog check only needs to run once, up front.
- **Transform menu, not more ContextBar buttons**: resize/flip/rotate
  moved into a proper dropdown menu (matching File/Edit/Palette/Window/
  Help's existing shape) rather than staying as inline `ContextBar`
  controls, so a 3×3 anchor-grid picker and per-action scope questions
  have room to exist as real modals instead of squeezed inline widgets.
- **Scope-modal gating, later revised**: the Transform menu's Flip/Rotate
  originally only opened a modal (asking "apply to all frames?") when a
  Draw-mode project had 2+ frames — otherwise it ran instantly, matching
  the old one-click `ContextBar` behavior. Mid-session feedback changed
  this: the user wants the modal to *also* ask which target (Canvas/Layer
  in Pixel tier; Canvas/Layer/Shape in Shape tier) to apply to, and
  confirmed this choice is meaningful even with a single layer, so the
  modal should now always open in Draw mode. Also confirmed Layer/Shape
  should gain 180°/CCW rotation to match Canvas/Glyph, so every target
  offers the same 5 operations — this part is implemented but **not yet
  committed or browser-verified** (see "Deferred / parked" below).

## Notable problem-solving

- The gradient-handle propagation bug (see above) was only found because
  manual browser verification checked the *exact* undo-step count after a
  drag, not just that the drag visually worked — a good example of why
  this project's "verify in the browser, not just unit tests" convention
  keeps paying off.
- A fresh Draw-mode project has **zero layers** until the first paint
  (Pixel tier creates layers lazily via `autoLayerSync`) — a test that
  read `canvas.layers[0].id` before painting anything failed with a
  generic "Cannot read properties of undefined" error; fixed by painting
  first. Worth remembering for any future store-level test that needs a
  `layerId` on a fresh project.
