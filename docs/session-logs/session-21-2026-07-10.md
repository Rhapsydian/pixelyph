## Work Completed

### Checkpoint 1 — Pixel paint-tool cluster (`a23f283`, `f77b3a7`, `06ca9ec`)

Brush width (1–8px), a v1 checkerboard dithering texture, and pixel-perfect
corner correction for pencil/line, plus bucket fill's tolerance and global
(non-contiguous) modes. Pencil gained path-following interpolation between
pointer samples (`computeLineCells`) to fix a latent fast-drag gap, with a
one-step buffer (mirroring Aseprite's approach) so a corner artifact can be
detected and dropped before ever being painted rather than needing to
"unpaint" it after the fact. Bucket fill's tolerance reused a newly
extracted `colorDistance.js` (pulled out of `importRaster.js`'s private
`hexToRgb`). New `ContextBar.jsx` two-column layout: existing controls
plus a border-separated group of tool-gated controls.

Tests: 338/338 → 368/368. Hit a real scare during manual verification —
simulating a multi-point drag via raw `PointerEvent` dispatch (no true
pointer-capture backing) corrupted browser state and made it look like a
single click had filled the entire canvas. Root-caused to the test
harness's synthetic events, not the shipped code, by re-verifying every
changed path via direct calls against the real store instead — all
correct. Lesson carried into every later checkpoint's verification.

### Checkpoint 5 — OS-clipboard image paste-in (`282992c`)

A `window` `paste` event listener in `SvgPixelEditor.jsx` reads
`clipboardData` for an image MIME type and routes it through the existing
decode+quantize pipeline, landing it as a floating selection (matching
internal Ctrl+V's drop-in point) via a new `pasteImageBlob` store action —
rather than `importRasterImage`'s own immediate full-canvas paint+commit.
Falls through untouched when the clipboard has no image, so internal
paste is unaffected (verified explicitly).

Verified in the web build via a synthetic `ClipboardEvent`: paste, move,
drop, undo (reverts paint+palette together), Escape-cancel (reverts the
palette merge too). The roadmap explicitly asked for Electron confirmation
too, which no available tool could drive directly — flagged that gap
honestly rather than assuming, and the user tested it manually themselves
after being offered the choice; confirmed working.

### Checkpoint 6 — Whole-canvas/layer/shape flip and 90° rotation (`2b21e47`, `7d93a49`)

The biggest and most novel checkpoint per the roadmap's own framing. Core
primitives (`flipPixelsH/V`, `rotatePixels90`) added to `Grid.js`, plus
shape-level ops (`flipGridH/V`, `rotateGrid90` — center-preserving) in the
same file. Layer/canvas-level ops in `Canvas.js` reposition each grid's
offset against the *canvas's* width/height, not its own, so a multi-shape
layer flips/rotates as one unit (matching Photoshop/Aseprite's "flip
layer") — derived and hand-verified the rotation's corner-mapping formula
(`new(nx,ny) = old(ny, H-1-nx)`) before writing any code, since this was
flagged as the one genuinely new piece of math. Glyph mode got its own
dedicated actions operating directly on the flat glyph buffer (mirroring
the existing `resizeActiveGlyph` pattern) rather than reusing the
layer/offset machinery, since a glyph has no layers of its own — rotation
re-crops/pads back to the font's `pixelsPerEm` behind a confirm, only
shown when that's actually lossy.

UI: `LayersToolbar` (`LayersPanel.jsx`) gained Flip H/V/Rotate 90° for
both the shape and layer action branches (three new icons in `icons.jsx`);
`ContextBar.jsx` got the same three for whole-canvas/whole-glyph, plus an
"All frames" checkbox honored by the layer/canvas-level actions (an
implementation-time UI-affordance choice the roadmap left open).

Tests: 371/371 → 385/385 (14 new, across `Grid.test.js`/`Canvas.test.js`/
`GlyphSet.test.js` — the primitive, shape-level center-preserving rotate,
layer/canvas-level offset-remap math against canvas dimensions, and
glyph-level re-crop behavior).
