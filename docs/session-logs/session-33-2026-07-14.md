## Key decisions and corrections (with rationale)

1. **Accepted transitional breakage rather than reordering checkpoints.**
   Checkpoint 1's model-layer changes alone (dropping `GlyphSet.kind`)
   turned out to break glyph-adding in the running app entirely — both
   `CharacterMapPanel`'s tab and `GlyphSetPanel`'s icon-add form are gated
   on `kind`, which no longer exists — plus left 6 tests red until
   checkpoints 5-6 caught up. Surfaced via `AskUserQuestion` rather than
   pushed through silently, since it conflicts with this project's usual
   "every checkpoint commit is independently verified working" norm. The
   user chose to proceed exactly as the plan doc sequenced it (tests
   green again at checkpoint 6, glyph-adding UI back at checkpoint 7/8)
   over reordering or bundling checkpoints to avoid the gap.
2. **Widened the selection alert's scope during checkpoint 8's wireframe
   review.** The plan doc originally excluded the empty-grid-alone case
   from the selection alert ("visually self-evident as a blank canvas"),
   showing it only for missing-codepoint-or-name. The user corrected this
   after seeing the wireframe: the alert should fire for any of the three
   caution-badge conditions (auto-assigned codepoint, missing name, **or**
   empty grid), with fixed text "Glyph missing codepoint, name, or
   content." — a real, deliberate deviation from the plan doc, not an
   implementation slip. Left a note in `BACKLOG.md` so later checkpoints
   don't reference the plan's original narrower wording.
3. **Bulk-Add button ships now, inert, rather than combined or omitted.**
   The plan splits the "Bulk Add…" button (home: `GlyphSetPanel.jsx`,
   checkpoint 8) from the modal it opens (`CharacterMapPanel.jsx`
   converted, checkpoint 9) across two checkpoints. Asked the user how to
   handle the gap; they explicitly wanted the button visually in place
   now, wired up next checkpoint, over combining 8+9 into one pass or
   leaving the button out until it's functional.
4. **Confirmed, not changed:** the unified `character` field (both the
   add-glyph form and the edit sub-panel) keeps using the existing richer
   `parseUnicodeInput` — literal paste, `U+xxxx`, numeric entities, and
   named HTML entities like `&hearts;` all still work, carried over
   unchanged from today's icon-kind Unicode input.
