## itch.io publishing — resolved via `butler push`

Downloaded `butler` (v15.29.0) from itch.io's own distribution
(`broth.itch.zone`) to `C:\Users\husbando\tools\butler\butler.exe` — kept
off the system `PATH` and invoked by full path, per the session's own
"avoid system-settings changes" instinct. Authenticated via `butler login`
(browser OAuth, credentials cached locally, no key ever handled directly).
Built with the existing `npm run build:itch` (session 36), then pushed with
`butler push dist rhapsydian/pixelyph:html5` — succeeded on the first try
(upload `#18342445`, build `#1797874`).

The live page itself 404'd to an anonymous visitor at first ("We couldn't
find your page") — turned out to be the project's itch.io visibility still
set to Draft from earlier testing, not a butler/build problem. Once the
user flipped it, the page loaded correctly. Verified the actual fix (not
just "the page loads") by reading the embedded game iframe's src
(`html-classic.itch.zone/html/...`) directly and checking network requests
for `assets/index-*.js`/`assets/index-*.css` — both `200`, no console
errors, full app UI rendered and interactive. This confirms the root cause
diagnosis from session 36: itch's own zip-unpacker mishandles
subdirectories, and `butler push` sidesteps it entirely by uploading files
directly via itch's API/CDN instead of a zip itch has to unpack itself.

**Resolution:** `butler push` is now the standing itch.io deploy method.
The flat-output workaround scoped in session 36 (`assetsDir: ''` +
flattening `manual/`/`samples/`) is unnecessary and was dropped from
`BACKLOG.md`. **New convention, at the user's explicit request:** an
explicit "push to GitHub" for this project now also means deploy to
itch.io (`npm run build:itch` + `butler push`) — documented in
`.claude/dev-session.md`'s commit policy and saved to cross-session memory,
since it's a standing behavioral rule, not a one-off.

## Screenshot capture — partially completed, handed off to the user

Attempted to capture an itch.io cover image (630×500) and a 5-shot
application showcase set (overview/gradients+glow, Layers panel, Glyph mode
+ Specimen Preview, Animation frame strip, Export panel). Hit three
compounding tooling problems:

1. **Browser pane `computer` screenshot/zoom timed out consistently**,
   same failure as session 36's note — confirmed not fixed, and not
   state-dependent (happened even against the plain running dev server,
   not just the itch.io page).
2. **Switched to Claude in Chrome** (user installed and connected the
   extension mid-session) — screenshot capture worked there, and one
   pixel-perfect 632×502 cover-image crop plus all 5 app screenshots were
   successfully captured and displayed. `resize_window` was unreliable
   though: a no-op while Chrome was maximized, and imprecise even after —
   requested 1280×800 landed at 1568×698 one time and 1156×615 another,
   with the app's own state (startup overlay) unpredictably resetting
   between tool calls, requiring re-navigation through the sample-project
   flow each time.
3. **The real blocker:** the screenshot tool's `save_to_disk: true` option
   does not write to any filesystem location reachable by this session's
   Bash tool — searched Downloads/Pictures/Documents/Temp/AppData broadly,
   twice, found nothing both times. Separately, captured images are not
   auto-displayed to the user in their chat view — they're nested inside a
   collapsible "Used N tools" chip the user has to manually expand to see
   and save. Neither of these was previously known; both are worth
   remembering for future sessions doing visual/asset work in this
   environment.

Given the friction, the user chose to capture the screenshots themselves
rather than keep fighting the tooling. Ended by giving the user a written,
navigable spec for the 5 recommended shots (exact File → Sample Projects /
panel-tab steps for each) instead of delivering image files directly.

## Problem-solving notes

- Browser pane (`mcp__Claude_Browser`) `computer` screenshot/zoom timeouts
  from session 36 are confirmed still present and not page-specific.
  Claude in Chrome doesn't have this problem reliably, though it does
  occasionally hit the same CDP `Page.captureScreenshot` timeout — a
  same-call retry consistently succeeded when it happened there.
- `save_to_disk: true` on the screenshot/zoom tools doesn't write to any
  location this session's Bash tool can reach — confirmed via two separate
  broad searches (background + synchronous) covering the full user
  profile. Tool-output images are also not shown to the user by default;
  they're nested inside a collapsible tool-use chip. Both are real,
  reproducible environment constraints, not one-off flukes — worth
  checking early in any future session that needs to hand the user actual
  image files, rather than rediscovering this from scratch.
- `resize_window` (Claude in Chrome) silently no-ops while the OS Chrome
  window is maximized; after un-maximizing it does resize but not
  precisely to the requested dimensions, and the app's client-side state
  reset unpredictably between some tool calls regardless of window size —
  cause not isolated (possibly Vite HMR, possibly extension
  reconnect/re-attach behavior).
