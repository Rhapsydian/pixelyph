## Phase 7: Animation

**Model:** `Canvas` gained `frameCount`, `activeFrame`, and `frameRate`, with every layer's `frames` array kept the same length as `frameCount` in lockstep — `addFrame`/`duplicateFrame`/`removeFrame` (`Canvas.js`) are the only places that length ever changes, mirroring the existing `growToInclude`/`resizeCanvas` pattern for "one shared invariant, enforced at a few call sites." Every place that used to hardcode `frames[0]` (`paintCell`, `colorAt`, `topVisibleLayerAt`, `mergeLayerDown`, `autoLayerSync.js`, `selection.js`) now reads the active frame instead. `frameCount` is undo-tracked content (like a resize); `activeFrame` is a working-session pointer (like `activeLayerId`), persisted but excluded from undo.

**UI:** `FrameStrip.jsx` (thumbnails via a new `composeFrameBody` primitive, add/duplicate/delete, frame-rate control) and onion skinning in `SvgPixelEditor.jsx` (adjacent frame(s) rendered behind the current one, tinted reddish/bluish via a CSS `filter` and faded via opacity).

**Exports:** `animatedSvg.js` (one `<g>` per frame, originally stepped via a single shared `@keyframes` rule using `steps(1,end)` and per-frame negative delays), `spriteSheet.js` (single-row PNG tiling + JSON metadata, reusing `rasterizeFrame` once per frame), `animatedRaster.js` (GIF via `gifenc`, split into a pure `encodeFramesAsGif` and a DOM-dependent `buildAnimatedGif` wrapper so the encoding logic itself is `node --test`-able).

**Bug found only by using it in a real browser** (matching every prior phase's experience): `gifenc`'s Node build (CJS, resolved via `package.json`'s `main`) and Vite's browser build (real ESM, resolved via `module`) have genuinely different export shapes — `import gifenc from 'gifenc'; const { GIFEncoder } = gifenc` worked under `node --test` but threw `GIFEncoder is not a function` in the actual browser. Fixed with a namespace import (`import * as gifencNs`) plus a runtime shape check (`gifencNs.GIFEncoder ? gifencNs : gifencNs.default`), which is correct in both environments.

**Testing:** 232 tests after this phase (from 201). The GIF tests needed a real, if minimal, GIF87a/89a walker (`parseGifFrames`) rather than the naive byte-pattern scan I wrote first — scanning for gifenc's Graphic Control Extension marker (`0x21 0xF9 0x04`) anywhere in the buffer risks a false-positive match inside LZW-compressed image data, which is exactly what happened once I tried to decode per-frame delay values out of specific byte offsets rather than just counting markers.

**Manual verification:** decoded the exported animated SVG/sprite-sheet PNG+JSON/GIF and confirmed their structure and dimensions matched expectations (this is also where the gifenc bug above was actually caught).

## Follow-on: Per-Frame Duration

User question: "Is it possible to set the duration of each frame?" — no, only a single canvas-wide `frameRate`. User asked to implement it.

`Canvas.frameDurations` (ms, one entry per frame) replaces the single rate as the actual timing source; `frameRate` becomes just the default duration a *newly-added* frame gets, not a retroactive rescale of existing ones. `addFrame`/`duplicateFrame`/`removeFrame` keep it in lockstep with `frameCount` the same way they already do for layers' `frames` arrays (duplicated frames copy the source's duration).

This forced a real rework of `animatedSvg.js`: the single-shared-`@keyframes`-plus-uniform-delay trick only works when every frame is the same length, so it became one `@keyframes` rule per frame, each sized to its own share of the total cycle and offset by the cumulative duration of every frame before it. The GIF and sprite-sheet exports needed much less — `gifenc`'s `writeFrame` already takes a per-frame `delay`, and the metadata just gained a `duration` field per frame entry (matching Aseprite's actual JSON convention) instead of one top-level `frameRate`.

Also hardened the GIF test file further here: my first attempt at verifying per-frame delay via the naive marker-scan approach gave a wrong answer (a false positive from pixel data, same class of bug as before), which is what prompted writing the real `parseGifFrames` walker instead of patching around it again.

**Manually verified** by setting frame 2 to 500ms and confirming all three exports reflected it (the animated SVG's CSS math, the GIF's decoded per-frame delay via the real parser, and the sprite sheet's JSON) — not just trusting the unit tests.

**Commit:** `79964b0`

## Follow-on: In-Editor Play/Pause Preview

User question: "Is there a way in-app to view the animation?" — no, only manual frame-by-frame stepping; the only way to actually see it animate was exporting and opening the result elsewhere. User asked to implement it.

A Play/Pause button in `FrameStrip.jsx`, backed by a `setTimeout` chain in `store.js` (not store state — a timer handle isn't serializable, same reasoning as the existing `autosaveScheduler`) that re-reads state on every tick, so it naturally adapts if frames are added/removed/re-timed mid-playback. Manually navigating to a frame pauses it (the "scrubbing takes back control" convention most video/animation UIs use), and every project-lifecycle action that swaps the current document (`newProject`, `closeProject`, `openAnyProject`, `resumeAutosave`) stops any running timer so it can't leak across projects.

**Verified live in the browser** by sampling the editor's actual rendered `<path>` data every ~90ms while playing and confirming it alternated between the two frames' real shapes in sync with their durations, then confirming Pause froze it exactly where it was.

**Commit:** `49aad80`

## Follow-on: Pause Without Painting

User questions: "Is editing disabled while the animation is playing? Is the play button disabled when there's only one frame?" — the second was already true (`disabled={canvas.frameCount <= 1}`), but the first wasn't: nothing guarded the paint/tool path against `isPlaying`, so painting during playback would land on whatever frame happened to be active at that instant (which keeps changing every tick) — or even split a single stroke across two frames if a tick fired mid-drag. User: "definitely pause the animation, but don't start the actual use of the tool."

`SvgPixelEditor.jsx`'s `handlePointerDown` now checks `isPlaying` first: if true, it pauses and sets a `suppressGestureRef` flag, returning before dispatching to any tool and without setting `isPointerDownRef` — which already makes `handlePointerMove`'s existing guard skip the tool for the rest of that gesture. `handlePointerUp` checks the same flag to skip its own (previously unconditional) tool call.

**Verification took a wrong turn worth recording:** my first few browser checks gave inconsistent results — sometimes the click paused-and-didn't-paint as expected, sometimes it looked like painting had happened. Chased this down rather than shipping on ambiguous evidence: the actual cause was two layered mistakes in my *test* methodology, not the app — (1) setting a frame's duration via a batch of stale DOM element references left one commit not actually landing, so real ~83ms ticks were firing throughout what I thought was a slow, controlled window; (2) once that was fixed, dispatching a synthetic `PointerEvent` with a fresh `pointerId` makes `element.setPointerCapture()` throw `NotFoundError` (no real browser ever establishes that pointer as "active"), which aborts `handlePointerDown` *before* reaching the new `isPlaying` check at all — a pre-existing property of the original handler, not something introduced by this fix, but invisible until a pointerId that wasn't already valid was tried. Reusing a `pointerId` already established by real interaction in the same page session made the synthetic events valid, and the fix confirmed cleanly: a click, and a multi-cell drag, across previously-blank cells during playback paused immediately and painted nothing.
