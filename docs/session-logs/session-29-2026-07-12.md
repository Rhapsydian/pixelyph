## Design process (plan mode)

Two Explore-agent passes (model layer, UI/interaction layer) mapped the
existing selection system and found it had accreted **four independent,
loosely-related concepts** across sessions 18-28: `selection`/
`floatingSelection` (marquee state), `activeGridId`/`activeLayerId`
(paint-target pointer, also driven by `selectMove`), `selectionScope` (the
"Select from:" dropdown), and `selectionKind` (a `LayersPanel.jsx`-local
row-click tracker). The scope flag's three values (`activeShape`/
`activeLayer`/`allVisible`) were read with genuinely different semantics by
different consuming functions â€” topmost-wins for Copy/Cut, "every matching
shape independently" for the new Transform path â€” with zero UI signal of
the split. That mismatch, not a simple logic bug, was the root cause of
"doesn't work correctly at all."

Research into prior art (Aseprite, Photoshop, Illustrator, Figma) confirmed
a pattern: raster/pixel-extraction operations are always single-layer
scoped in real tools (never a silent cross-layer flag), while object-level
transforms treat multi-object behavior as an explicit, separate choice
(Illustrator's "Transform Each" vs. plain Transform). Several rounds of
user feedback then converged the design further, each building on the
last:

1. Keep `marqueeSelect` and `selectMove` as two separate tools â€”
   `selectMove`'s behavior was already correct, just confusingly named
   (renamed to `targetMove`).
2. Remove `allVisible` as a scope option entirely, everywhere (Copy/Cut/
   Paste and Transform alike) â€” matches Aseprite/Photoshop's convention of
   never doing cross-layer marquee operations.
3. Locked and hidden shapes are fully immune to every scope-consuming
   operation, uniformly â€” including `activeShape`, which checked neither
   before.
4. The user's own reframing, which turned out to collapse the whole
   design: transforming a selected area is conceptually *cut it out (per
   shape, preserving style) â†’ operate on it â†’ paste it back*, and â€” the
   key insight â€” that "paste" step must require an explicit finalize
   (Enter/click-outside/tool-switch), reusing `floatingSelection`'s
   existing interaction exactly, the same way a marquee drag-move already
   works. Once Transform *also* goes through a pending-then-finalize step,
   **Move and Transform become the same underlying operation.**
5. Paste-in-place: pasting content just copied/cut from the same document
   should land back at its original position, not recenter.
6. (Scoped but deferred to checkpoint 5, not yet built): external
   multi-color raster paste on Shape tier should offer "multiple shapes by
   color" (today's default) or "single shape" (a union, for importing a
   silhouette/mask) as an explicit choice.

### Checkpoint 3 â€” `floatingGridSelection`: unify Move and Transform (`4a9f783`)

The core of the session. New machinery in `selection.js`
(`liftGridSelection`/`moveGridSelectionBy`/`transformGridSelection`/
`finalizeGridSelection`/`clearGridSelectionSource`/
`pruneEmptyGridsForSelection`/`buildFloatingGridPreviewDoc`/
`buildGridClonesByColor`) replaces both the old instant-commit
`transformSelectionRegion` (deleted) and, for Shape tier only, the old
flat-cell `liftSelection`/`floatingSelection` path that `marqueeSelect.js`
used for drag-move. Real, detached Grid clones carry `originGridId`
(which real grid to write back into, `null` for copy-drag/paste content)
and an `originSnapshot` (frozen at lift time, used only to clear the
source on finalize) â€” `canvas.layers` is never mutated until finalize, so
Escape/Cancel is a true no-op instead of a history revert.

Wired into: `marqueeSelect.js`'s lift/move (Shape tier only â€” Pixel
tier/Glyph mode keep the flat path, no style to lose there),
`transformSelectionInStore` (Transform-menu flip/rotate now lifts-if-needed
and applies to whatever's pending, staying pending â€” no longer a special
instant-commit case), `copySelection`/`cutSelection` (Shape-tier clones,
`clipboard.kind: 'grid'`), `pasteImageBlob` (external paste groups by
color via `buildGridClonesByColor`), and `SvgPixelEditor.jsx`'s render
(`buildFloatingGridPreviewDoc` substitutes each clone for its real grid so
the actual styled/gradient content previews live, not a flattened
approximation) plus its Enter/Escape/click-outside/tool-switch/arrow-nudge
handling.

Tests: 441/441 â†’ 452/452 (11 new model-level tests covering per-shape

`flipSelectionH` test in particular now asserts the transform stays

**Verified live, the whole point of the session**: two shapes (one flat

## Notes

- **The Browser pane automation tool has a few sharp edges worth knowing
  up front next time**: (1) its `key: "Return"` action does not map to the
  `evt.key === 'Enter'` check most web apps use â€” dispatching a real
  `KeyboardEvent('keydown', {key: 'Enter'})` via `javascript_tool` works
  reliably where the `computer` tool's own key action didn't. (2)
  `left_click_drag` did not reliably fire intermediate `pointermove`
  events for a *brand-new* marquee-select drag over empty canvas in this
  session (a single click's `pointerdown` alone worked fine and created a
  1Ă—1 selection) â€” drags that *moved an already-pending floating
  selection* worked without issue throughout, so this looks scoped to the
  bare "start a new rect selection" gesture specifically, not a systemic
  tool failure. (3) `read_console_messages` appears to retain a
  non-clearing buffer across navigations within a session â€” several
  "errors" investigated turned out to be identically-timestamped stale
  HMR-reload noise from earlier edits, not live problems; a fresh
  functional check (screenshot, interaction, DOM read) was the reliable
  signal each time, not the console log alone. (4) `computer` click/drag
  coordinates are in *screenshot*-pixel space, not real browser-viewport
  space â€” `getBoundingClientRect()` reads need dividing by the
  screenshot-to-viewport scale factor (~1.53 in this session's 800Ă—522
  screenshot vs. 1222Ă—798 real viewport) before comparing to click
  coordinates.
- **The redesign fully replaced session 28's `transformSelectionRegion`**
  rather than building alongside it â€” worth remembering that checkpoint 2
  quietly retired an entire function (and its dedicated test coverage)
  that had shipped (uncommitted) only one session earlier. Nothing wrong
  with that here â€” it was correctly identified as superseded, not
  orphaned â€” but it's a reminder that "uncommitted work from last session"
  isn't necessarily stable ground to build the next session's plan on
  top of, especially when the user asks for a from-scratch redesign.
- **Committing mid-redesign, before the plan is fully shipped, was a
