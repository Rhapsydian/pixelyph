### Refinement — off-canvas handle visibility (`c1a9562`)

User reported controls become invisible once dragged outside the canvas.
Root cause: the outer `<svg>`'s default (browser UA-stylesheet)
`overflow: hidden` clips anything drawn past its `viewBox`. Fix: nested the
artwork + non-interactive overlays in an inner, still-clipped `<svg>`,
while the outer `<svg>` (where handles render, as direct children) is
`overflow: visible` — preserves the existing "off-canvas content stays
hidden" convention for artwork while keeping interactive handles grabbable
past the canvas edge, up to the editor panel's own boundary.

floor) added to `gradientHandleGeometry.js`. New `GradientRadiusHandle` +
`GradientRadialHandle` (center + radius, connecting spoke). Checkbox gate
extended to `radial-gradient`. Verified center/radius drag live-updates the
gradient correctly, radius floor holds when dragged past center.

### Checkpoint 5 — radial focal point (`e94e7ce`)

`fx/fy` schema (defaults to `cx/cy`, matching SVG's own default so old
saves render unchanged). `serializeFill`'s radial branch passes `fx/fy`
through. `clampPointToRadius` keeps the focal point inside the radius.
`GradientPointHandle`'s diamond variant (rendered last/on top so it's
grabbable even coincident with the larger center circle).

### Refinement — focal-point clamp margin (`66c8661`)

User reported odd rendering with the focal point right on the radius edge
— a known SVG quirk where a radial gradient's focal point exactly on the
boundary produces a visible seam in some browsers. `clampPointToRadius` now
caps at 97% of `r` instead of exactly `r`.

### Refinement — focal point follows center/radius changes (`e12fc2e`)

User suggestion: an explicitly-set focal point should translate with the
center and rescale proportionally with the radius, instead of staying at a
fixed absolute position (which could silently drift outside the radius or
just end up somewhere unintended). Added `translateFocalPoint`/
`rescaleFocalPoint` (no-ops when the focal point was never explicitly set —
it already tracks center via its `fx ?? cx` fallback). Applied to *both*
interaction paths for consistency: the on-canvas drag handles
(`GradientRadialHandle.jsx`) and the modal's numeric cx/cy/r inputs
(`GradientEditorModal.jsx`), which had the same staleness problem. Proportional
rescaling was proven to automatically preserve the earlier clamp margin
(no re-clamping needed) — verified numerically, not just visually: the
offset-to-radius ratio matched exactly (0.667) before and after a radius
change, in both the modal and on-canvas paths.

## Notes

- **SVG's default root `overflow` is `hidden`, not `visible`** — easy to
  forget since most block-level elements default the other way. Any future
  on-canvas control that needs to extend past the artwork's own bounds
  needs this same nested-clipped-inner-svg-plus-visible-outer-svg pattern,
  not a one-line CSS tweak.
- **`cloneLayerStyle` (`src/model/Canvas.js`) reconstructs a fresh
  `{fill, stroke, effects}` object** — it silently drops any other
  top-level field spread onto its input. Caught during this session's
  *planning* pass for the palette-name-prompt item (not yet implemented):
  naively doing `addPaletteStyle({...grid.style, name})` would have the
  `name` field silently stripped by `addPaletteStyle`'s existing
  `cloneLayerStyle(styleValue)` call before it ever reaches the palette
  entry. The plan doc adds `name` as a second parameter instead, applied
  after cloning.
- **Browser-pane screenshot flakiness, not an app bug:** the `computer`
  tool's `screenshot` action timed out several times in a row during radial
  handle testing while every other action (clicks, `read_page`, console
  reads) kept working — confirmed the app itself was healthy via DOM state
  and a fresh-tab console check each time, then screenshots resumed working
  again unprompted. Also hit a stale-console-history case: an `[hmr]
  Failed to reload` error kept showing up in `read_console_messages` well
  after the file was fixed, because it was logged once during a genuinely
  broken *intermediate* save state between two sequential edits — a fresh
  tab's clean console was what settled it. Worth remembering: when a
  console error's timing is ambiguous, check a **fresh tab**, not just the
  existing one's accumulated history.
- Session flow deviated from a strict "implement three items" plan mid-way
  — after the plan was written and approved, the user redirected the frame-
  reorder design (per-card → shared toolbar) and added the horizontal-scroll
  requirement *while still in plan mode*, before any Checkpoint 2 code was
  written. The plan file was revised in place across several rounds before
  the user chose to bank it for next session rather than execute — a real
  example of plan mode absorbing scope changes for free, with zero
  wasted implementation.
