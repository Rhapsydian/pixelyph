## Work Completed

### Plan-mode scoping (no commit — this is the process that shaped the
eventual spec)

Started from the original sub-step 2 framing: add a radial `cx/cy/r`
on-canvas handle to match the shipped linear-angle handle. An Explore
agent pulled the full context (the shipped `GradientAngleHandle.jsx`/
`gradientHandleGeometry.js` implementation, the radial fill data shape,
`serializeFill`'s gradient serialization, existing on-canvas handle
patterns) before any design work started.

The scope then grew through several rounds of pushback and follow-up
questions, each one changing the plan materially:

1. **Corrected a wrong claim.** The initial plan characterized linear
   gradients as "1 degree of freedom" (angle only). Called out that SVG's
   `linearGradient` actually takes a full `x1,y1,x2,y2` endpoint pair (4
   DOF) — the shipped handle just fixes center and magnitude, discarding
   3 of those. Same issue on the radial side: SVG's `radialGradient` also
   supports a focal point (`fx,fy`) distinct from center, not just
   `cx,cy,r`.
2. **Scope question, asked via `AskUserQuestion`**: keep radial minimal
   (`cx,cy,r` only), add the radial focal point, or redo linear's model
   entirely around free endpoints. Answer: both gradient types should get
   real control — keep the existing angle mode *and* add a free-draggable
   linear-endpoints mode; radial gets `cx,cy,r` drag handles.
3. **Control placement question**: should the enable/disable toggle
   (currently a checkbox in the Style tab's Fill box) move somewhere more
   discoverable, like a toolbar button, since a user focused on the Style
   tab might not notice the canvas update? Concluded the Fill-box
   placement is actually correct — it's contextual to the shape/fill being
   edited — and confirmed no relocation.
4. **Wireframes.** Per this project's dev-session convention (UI changes
   need mockups before go-ahead), produced ASCII wireframes for the
   gradient editor modal in each mode and the on-canvas handle appearance
   for all three gradient states (linear angle, linear endpoints, radial).
5. **Focal point scope question**: worth including this session, given
   it's a real but separate SVG feature (off-center highlight look, needs
   a clamp-to-radius rule and a visually distinct handle so it isn't
   invisible under the center handle at its default coincident position)?
   Answer: include it now, reusing the same generic draggable-point
   component being built for the other handles.
6. **UX bug caught by inspection, not by testing**: the shipped linear
   angle handle's spoke length scales with the shape's bounding box (it's
   pinned to the bbox edge), so a tiny shape gets a barely-grabbable
   handle and a large one gets an oversized spoke. Added a fix — pivot at
   the bbox center, project outward by a fixed on-canvas length instead of
   a bbox-scaled fraction — to the plan.
7. **Reuse instruction**: don't add a new control for the generalized
   gate — rename the existing checkbox ("Show angle handle on canvas" →
   "Gradient fine controls") rather than introducing a second toggle.
8. **Reset button, raised as an exploratory question** ("would it be
   worth..."): given the control surface had grown to mode + endpoints +
   focal point + clamp behavior, agreed a small Reset button (position/
   geometry fields only, stops untouched) was worth the modest cost.
9. **Final call**: given all of the above, the user asked to split the
   design into checkpoints and defer implementation entirely to next
   session, rather than build any of it now.
