## Goal

User-reported bug: in an animation where every frame is 300ms except one
at 600ms, the frame after the 600ms one appeared too early, leaving a
visible gap right before the 600ms frame played.

## Work Completed

### Root-cause diagnosis (no commit)

Read `src/export/svg/animatedSvg.js`'s `buildAnimationCss` — each frame's
`<g>` toggles opacity via its own `@keyframes` rule plus a CSS
`animation-delay`, sized/offset from that frame's own share of the total
cycle. The delay was computed as `-(cumulativeMs / 1000)`, where
`cumulativeMs` is the summed duration of every frame *before* frame *i*.

Before touching code, wrote a standalone Node simulation (not part of the
test suite) that replicates the actual CSS animation semantics — for
100,000 sample points across one full cycle, compute which frame the
generated delay/keyframe values make visible, and compare against the
expected schedule from the raw durations. Results: the old formula produced
50,000/100,000 mismatches for 4 *equal*-duration frames, and
100,000/100,000 for the reported uneven case — it only happened to work for
a trivial 2-equal-frame case, where `cumulativeMs` and `totalMs -
cumulativeMs` coincide by symmetry. This is why it shipped unnoticed: the
bug affects any animation with 3+ frames (frames land in shuffled
real-time slots), but it only becomes a *visible gap* once a duration
actually differs from its neighbors.

Correct delay: `-((totalMs - cumulativeMs) / 1000)` — a negative
`animation-delay` means "act as if the animation already ran this long,"
so the delay needs to reflect how far the frame's slot sits from the
**end** of the cycle, not the start. Verified the fixed formula against
the same simulation: 0 mismatches (aside from one floating-point boundary
sample, negligible).

### Fix (`ee16e90`)

One-line change in `buildAnimationCss` (`src/export/svg/animatedSvg.js`).
No other changes needed — `onPercent`'s formula and the keyframe/class
rule strings were already correct.

### Tests

`test/export/svg/animatedSvg.test.js` had two tests hard-coding the old
(buggy) delay values as expected output — updated both to the corrected
values rather than leaving them passing on stale assumptions. Added a new

that parses the generated CSS's actual delay/duration values and simulates
the resulting on/off timeline directly, asserting each frame is visible at
its own midpoint — so this fails structurally if the sign/formula
regresses again, not just if the literal numbers change.
