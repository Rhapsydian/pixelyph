## Key decisions and corrections (with rationale)

1. **`GlyphSetPanel.jsx`'s add flow was reworked a second time**, before
   finishing checkpoint 8's leftover verification, after the user reviewed
   the shipped UX directly. The "Create" button and top add-glyph form
   were scrapped for a `+` button (adds one bare glyph immediately,
   selecting it) plus a single Character/Name field pair that always edits
   the active glyph — replacing the plan doc's separate add-form/edit-
   sub-panel split. Two follow-on corrections landed in the same pass: a
   fresh no-preset project now seeds one bare glyph *and* selects it
   (matching what `+` produces, rather than leaving nothing active), and
   the `+` button was resized from a mismatched 20px square to 34px,
   matching the sort-toggle group's actual rendered height.
2. **Checkpoint 9 (Bulk-Add modal) shipped close to the plan doc.**
   `CharacterMapPanel.jsx` turned out to already be fully dead code (its
   `glyphSet.kind !== 'characters'` guard was permanently true since
   checkpoint 1 removed `kind`, and it called `assignCodepoint`, an action
   already retired in checkpoint 2) — rewritten wholesale as
   `BulkAddGlyphsModal.jsx`, wired to `addGlyphsFromPreset` (which already
   existed and needed no changes).
3. **Checkpoint 10's glyph display-color control needed a bigger touch
   than "small, isolated, low risk"** as the plan doc assumed. Real
   glyph-mode paint colors are baked in at paint/reconstruction time (the
   model's `pixels` buffer is a boolean bitmap; `GLYPH_FILL` is only ever
   applied when `glyphToCanvas` rebuilds the editing surface), so making
   the picker take effect required threading `glyphDisplayColor` through
   both `paintCellLive` and a new `buildGlyphCanvas` helper used at every
   `glyphToCanvas` call site (11 of them) — not just adding a swatch.
   `setGlyphDisplayColor` was further corrected mid-session to recolor the
   *currently displayed* canvas immediately, not just future rebuilds,
   after manual testing showed picking a new color did nothing until the
   next paint/selection/undo.
4. **Checkpoint 11 (Specimen Preview) picked up the most follow-up
   refinement of any checkpoint**, all from direct user testing after the
   initial redesign shipped:
   - The insert-glyph swatch row needed its own scroll cap (a full Basic
     Latin preset is 95 swatches) instead of pushing the color row/preview
     box down; the panel itself gained `FrameStrip`'s drag-to-resize
     pattern plus a collapse toggle.
   - The caution badge's logic was corrected: it now only fires when a
     glyph has *neither* a real codepoint *nor* a name (not just one of
     the two) — a real typed character with content but no name was still
     showing the badge under the old any-one-of-three-conditions rule.
   - `iconTilePadding` was renamed `horizontalPadding` and — the bigger
     change — widened to apply to *every* glyph, not just auto-assigned
     ones, after the user found it had no visible effect on a typed Latin
     glyph (correct per the old scoping, but confusing without that
     context). `glyphMetrics()` now adds the padding on top of whichever
     base bearing/advance applies; old saves carrying the former key get
     it copied across on load.
   - The preview's row-to-row vertical gap was a flat, metric-unrelated
     2px CSS gap — real testing (two rows of the same tileable glyph)
     showed horizontal tiling could go fully seamless while a vertical
     seam remained; removed entirely.
   - The preview box was given a one-line minimum height so the
     (now-shrinkable) insert-glyph row can compress under space pressure
     without squeezing the actual preview out.
   - Two `ColorAlphaInput` swatches (ContextBar's glyph display color, the
     preview color picker) sat flush against their row's top edge instead
     of centered against taller sibling icon buttons — fixed with
     `alignItems`/`alignSelf: 'center'` at the two call sites.
5. **Checkpoint 12 (docs pass) retired `docs/glyph-mode-unification-plan.md`**
   entirely, folding its content into a new `BACKLOG.md` Shipped entry
   that documents the real deviations from the plan doc (all of the above,
   not just checkpoint 8's session-33 alert-widening) rather than treating
   them as slips. `docs/features.md`, `docs/data-model.md` (new "Glyph
   model unification" section), `public/manual/glyph-mode.md` (full
   rewrite), and `public/manual/getting-started.md` were all updated to
   describe the unified model with no reference to the retired
   character/icon split.
6. **A further round of user-driven polish landed after checkpoint 12**,
   once the whole refactor was nominally "done": a stale glyph-editor
   placeholder string left over from the pre-Part-A Create-button flow;
   Specimen Preview's insert-glyph swatches now insert at the textarea's
   actual cursor position instead of always appending (the existing
   `diffColors()` diff already handled a mid-string insertion correctly —
   only the insertion point itself needed fixing); and `demoHtml.js`'s
   bundled specimen preview was brought to parity with the in-app panel —
   insert-at-cursor swatches, the same hybrid per-character color model
   (native color picker + "Apply to all"), rendered through the real
   embedded `@font-face` font so multi-line is just `white-space:
   pre-wrap` with no manual metrics math needed, plus `preventScroll:
   true` on the swatch-click focus call after the user noticed clicking
   through the swatch grid kept yanking the page back up to the textarea.
   The demo's old tiling-test strip was dropped entirely per explicit
   feedback, rather than kept alongside the new color model.
7. **A user-manual rewrite for fresh users (no historical/legacy framing)
   was requested, scoped, then explicitly deferred** — the user chose to
   hold off and wrap the session instead of starting it. Left as
   unstarted work, not abandoned; whoever picks it up next should treat it
   as its own pass across every `public/manual/*.md` page, not just
   `glyph-mode.md` (which already got a from-scratch rewrite this session
   and may need a second look against a "no history at all" bar).
