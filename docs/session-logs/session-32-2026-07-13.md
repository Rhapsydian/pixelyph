## Goal

Kicked off via `/dev-session` with an explicit "planning and research
session only, no implementation" instruction. `BACKLOG.md`'s Open section
had several existing discussion topics (multi-select interaction, masks,
AI-agent scriptability, `.pixelyph.svg`), but the user redirected scope
entirely: overhaul Glyph mode's interface by merging the "character font"
and "icon font" sub-modes into one unified glyph type.

## Research

An Explore agent read `src/model/GlyphSet.js`, every file in `src/ui/glyph/`,
`src/export/font/*`, the New Project wizard in `App.jsx`, and
`docs/data-model.md` to establish ground truth: the `Glyph` data shape was
already fully unified between the two kinds — every character/icon
divergence was a local `if (kind === ...)` branch scattered across ~8 files,
not a real structural split. That made a genuine merge tractable rather than
a data-model rewrite.

A Plan agent then produced a first-pass design (drop `kind`, key every glyph
by codepoint with a PUA fallback, branch export metrics on codepoint range
instead of `kind`), which surfaced several judgment calls that got kicked
back to the user via `AskUserQuestion` rather than decided unilaterally
(the `glyph.unicode` field's fate, sort-order preservation, whether
`CharacterMapPanel` needed relabeling, a migration edge case).

## Key decisions and corrections (with rationale)

The user's answers substantially expanded and corrected the design across
three review rounds before approval:

1. **Full merge, no `kind` anywhere.** Every glyph can freely have a real
   typed character, a name, both, or neither — no project-level lock chosen
   at creation. A real typed/pasted character (e.g. pasting ❤ or typing
   `&hearts;`) becomes the actual codepoint key everywhere, retiring
   `glyph.unicode` as a separate decorative field (it was already
   confirmed dead at every export/render path except the UI).
2. **No "phantom slot" placeholders.** Corrected an early draft that treated
   the empty-glyph watermark as an addition to today's ghost-cell browsing
   behavior — the user clarified it should *replace* it: `GlyphSetPanel`
   only ever shows glyphs that have actually been added. Consequence: both
   the New Project wizard's initial-preset choice and the new Bulk-Add
   modal must eagerly create real, empty-grid glyphs with codepoints set,
   not just seed a browsing grid.
3. **Specimen Preview redesign**, prompted by the user flagging it as
   unwieldy once mixed-content glyph sets are possible: multi-line layout
   using real font metrics (the user's own call after a tight-tile-grid
   alternative was raised — gapless tiling is a font-metadata concern to
   document, not a special preview mode) and a hybrid color model (one
   global picker stamps newly-added glyphs; existing ones keep their
   locked-in color until an explicit "Apply to all").
4. **Model-only background/foreground layers.** The user's forward-looking
   idea for CSS `::before`/`::after` layered icon fonts got corrected twice:
   first from a single new buffer to two independent ones
   (`backgroundPixels` + `foregroundPixels`, three total layers), then
   scoped down to explicitly **no UI at all this pass** — pure data-shape
   groundwork for a fully separate future session.
5. **New, unscoped asks surfaced during review**: a Glyph-mode-only
   "display color" canvas preference (next to the grid toggle, display-only,
   not exported), and a request that Glyph mode's selection/transform match
   Draw mode's floating-selection-requires-finalize behavior exactly —
   flagged as needing explicit verification (not assumed) since `SvgPixelEditor.jsx`
   already appears to share the mechanism structurally between modes.
